## §0: Why this spec exists

The `concept:*` hub as built today is an *operation* hub. Its nodes are algebraic operations: `concept:add`, `concept:conditional`, `concept:while`, `concept:seq`, `concept:return`, `concept:call`, `concept:eq`, and the rest of the imperative core. A language's surface algebra desugars onto its core (the 2026-05-11 spec), the core renames onto `concept:core` (the cross-language-equivalence explanation), and `provekit migrate A → B` transports terms over that core. That machinery is real and it handles `1 + 1` and `if c { ... } else { ... }` and `while c { ... }` correctly.

It does not yet handle real programs. Real programs are full of dynamic dispatch, closures, exceptions, references and aliasing, iterators, generic instantiation. None of those is a single algebraic operation. A C method call through a vtable is `p->vtbl->method(p, args)`: a struct whose field points at a function-pointer table, indexed by a compile-time-or-link-time-fixed offset. A Python method call is `obj.method(args)`: a name-lookup walk over namespace dicts, `type(obj).__dict__` and the MRO. **Same idea, call the method appropriate to the receiver's runtime type, radically different realizations.** Neither language has "dynamic dispatch" as a single op. C *realizes* it via a vtable-indirection *pattern*. Python *realizes* it via a dict-lookup *protocol*. So the hub needs nodes that are *concepts*, patterns and protocols and capabilities, not just operations. The operation layer is the special case where the concept happens to be an op.

This spec defines that abstraction layer: the tier model of the hub (§1), what an abstraction-concept node is and its schema (§2), `concept:dynamic-dispatch` worked rigorously for the {C, Java, Python} validation trio and sketched for the other languages (§3), the other core abstraction-concepts sketched (§4), the mechanics, which are the operation-layer mechanics one tier up (§5), what it costs and what it is worth and the hard parts named honestly (§6), the relationship to the rest of the substrate (§7), what is already built versus a follow-up and the implementation scope (§8 to §9), and the closing principle (§10).

### §0.1 The premise, stated plainly

**There is no general-purpose target language that cannot express a well-formed concept.** This is Turing-equivalence at the abstraction level. Just as "Go has no ternary" does not mean "ternary cannot go to Go" (it desugars to `var t T; if c { t = a } else { t = b }`), "Python has no vtables" does not mean "dynamic dispatch cannot go to Python" (it is the dict-lookup protocol). Every general-purpose language realizes every well-formed concept by encoding it in what it *has*, with a refinement-and-loss profile. The only languages that genuinely cannot are sub-Turing, or so impoverished they lack the building blocks. ProvekIt is not targeting those.

Corollary, and a useful design criterion to state as a first-class rule: **a candidate target language is admissible iff every core concept, operation-layer and abstraction-layer, has a possibly-lossy-but-characterized realization in it.** A language that genuinely fails that is not a target. That is a signal about the language, not a gap to live with.

And: **the canonical form of an abstraction is the concept node itself.** For dynamic dispatch, neither C's vtable nor Python's dict is the canonical thing. The canonical thing is `concept:dynamic-dispatch`, and every language, including the one the program was written in, is a *lossy projection* of it. The hub node is its own thing: not "C's version with a `concept:` prefix," not "Python's version generalized," but the abstraction, with the languages all downstream of it. This is paper 17's name-by-vector at the abstraction tier: the shape's true name is its CID, every language is a dialect.

## §1: The tier model of the `concept:*` hub

The hub has layers. Each layer's nodes decompose into the layer below.

### §1.1 The operation layer

Algebraic ops, one algebraic function each: `concept:add`, `concept:conditional`, `concept:while`, `concept:seq`, `concept:return`, `concept:call`, `concept:assign`, `concept:eq`, `concept:deref`, `concept:member`, `concept:index`, the arithmetic and comparison families, `concept:throw`. This is the imperative core. Everything the transport work so far is about lives here. A language's surface algebra desugars onto its core (2026-05-11 §1), the core renames onto `concept:core`, and term transport runs over `concept:core`. The operation layer is closed and plateaus: paper 16 Lemma 2, the alphabet plateau, says there are finitely many irreducible computational ops.

### §1.2 The abstraction layer

Patterns, protocols, capabilities: `concept:dynamic-dispatch`, `concept:closure`, `concept:exception`, `concept:reference` / `concept:aliasing`, `concept:iterator`, `concept:generic-instantiation`, and the ones we have not named yet. A node here is not an op. It is *the idea* of a capability. It does not have a direct `<lang>:op → concept:abstraction` morphism, because no language has the abstraction as a primitive op. Instead it has, per language, a *realization*: a way that language encodes the abstraction in what it *does* have, expressed as a desugaring of the abstraction node into an operation-layer term over that language's core. C's `concept:dynamic-dispatch` realization desugars to a `concept:deref` + `concept:member` + `concept:call` term over the operation-layer ops. Python's desugars to a `concept:dict-lookup` (a new op-layer hub mint this spec calls for, see §1.4 and §8) + `concept:call` term. That operation-layer term then transports via the *existing* operation-layer morphisms. So the abstraction layer **sits on top of** the operation layer. An abstraction-realization is: desugar the abstraction node into this language's operation-layer term, then the operation-layer machinery handles the rest.

The abstraction layer also plateaus, and it is worth stating why, because that is the load-bearing reason the catalog is not open-ended: there are finitely many recurring *abstractions* the way there are finitely many recurring *ops*. "Call the method appropriate to the runtime type," "a function plus its captured environment," "non-local transfer of control with a payload," "a name bound to a mutable cell shared by reference," "a uniform stepping protocol over a sequence," "a type-parametric definition specialized at a point of use": these are the cluster centers in abstraction space, the Kohonen-map analogue of the concept-shapes node table one tier up. New language mints mostly form new *realizations* of existing abstractions, not new abstractions, the same way new language mints mostly form new core *terms*, not new core *ops*.

### §1.3 The paradigm layer (gestured at, not built)

Above the abstraction layer there is plausibly a paradigm layer: OO, functional, actor, dataflow. An OO "class" is a bundle of `concept:dynamic-dispatch` + `concept:reference` + a record-with-named-fields + a constructor protocol. A functional "ADT match" is `concept:closure` + a tagged-union destructure. An actor is `concept:closure` + a mailbox protocol + an event loop. This spec names the layer so the tier model is honest about where it stops. It does not build it. The abstraction layer is the next rung; the paradigm layer is the rung after.

### §1.4 How the layers relate

An abstraction-concept node *desugars into* a per-language operation-layer *term*. The desugaring is exactly a desugaring in the 2026-05-11 sense: a homomorphism of the term algebra that preserves the abstraction's contract `wp` modulo the loss record (§2.4). C's `concept:dynamic-dispatch → vtable-struct-deref-and-call`; Python's `concept:dynamic-dispatch → dict-lookup-over-MRO-and-call`. The realization term lives in the *language's* operation algebra, so it then transports via the operation-layer morphisms already minted (LSP, the cross-language-equivalence explanation, PTP §2).

Which operation-layer terms exist as the *targets* of these desugarings may itself drive new operation-layer hub ops. The vtable realization wants `concept:deref` + `concept:member` + `concept:call`, all of which exist. The dict realization wants a `concept:dict-lookup` (a dictionary lookup by key with a present/absent result) and arguably a `concept:hashtable-lookup` (the runtime-dispatch-table variant), neither of which is in the hub today. The prototype-chain realization wants a `concept:proto-walk` (walk a linked chain of property maps until found). These are follow-up op-layer mints, listed in §8. The point: the abstraction layer does not invent new *layers* of machinery, it consumes the op layer, but extending it surfaces precise requests to extend the op layer too.

## §2: What an abstraction-concept node is

### §2.1 The `ConceptAbstractionMemento`

A `ConceptAbstractionMemento` is a hub node at the abstraction tier. It aggregates: the abstraction's identity and slot structure; a *contract* describing the behaviour the abstraction guarantees, expressed as a `wp_rule`-style formula in the 2026-05-13 `wp`-formula language (§2.3); and a list of *realizations*, one or more per language, each a `RealizationDesugaringMemento` (§2.2). It is content-addressed like everything else: its CID is `BLAKE3-512(JCS(<canonical form>))` via `provekit-canonicalizer`, over its identity, slots, sorts, contract, and the list of realization CIDs (the realizations are referenced by CID, not inlined, so a realization can be added without re-minting the node, the same pattern desugaring-equation sets use).

```cddl
; Imports:
;   ir-formula      ; from 2026-04-30-ir-formal-grammar.md, extended per 2026-05-13 §2.3
;   arity-shape     ; from AMP / LSP
;   sort-ref        ; a SortMemento CID
;   loss-record     ; from 2026-05-14-transport-gap-and-partial-morphism-protocol.md (PR #616),
;                   ;   extended here with a fifth dimension `structural_divergence` (§2.4)

concept-abstraction-memento = {
  kind:          "concept-abstraction",
  operator:      tstr,                 ; e.g. "concept:dynamic-dispatch"
  tier:          "abstraction",        ; pins the layer; "operation" for op-layer nodes
  slots:         [+ abstraction-slot], ; named, ordered
  formal_sorts:  [+ sort-ref],         ; one per slot, same order
  result_sort:   sort-ref,
  contract:      ir-formula,           ; the wp_rule of the abstraction; §2.3
  ? contract_note: tstr,               ; human-readable; non-load-bearing; MUST be omitted when absent
  realizations:  [+ cid],              ; CIDs of RealizationDesugaringMemento, >= 1; the catalog grows here
  ? superseded_by: cid,                ; if a successor abstraction node refines this one
  ? refines:     cid                   ; LSP §4.4 successor-mint chain
}

abstraction-slot = {
  name:          tstr,                 ; e.g. "receiver", "method_name", "args"
  ? variadic:    bool                  ; default false; e.g. "args" is variadic
}
```

### §2.2 The `RealizationDesugaringMemento`

A realization is a desugaring of the abstraction node into one language's operation-layer term, plus the loss record that desugaring incurs, plus the discharge receipt certifying it is exact-semantics modulo that loss record. It is a `DesugaringEquationMemento` (2026-05-11 §2) elected into the `abstraction-realization` role, with three additive fields.

```cddl
realization-desugaring-memento = {
  kind:                "equation",                  ; unchanged: it IS a desugaring equation
  fn_name:             tstr,                          ; e.g. "concept:dynamic-dispatch->c11:vtable-indirection"
  formals:             [+ tstr],                      ; the abstraction's slot variables
  formal_sorts:        [+ sort-ref],
  ? pre:               ir-formula,                    ; side condition: WHEN this realization is admissible
                                                      ;   (e.g. "static dispatch table" for the C vtable case)
  post: {                                             ; the equation lhs = rhs
    lhs:               ir-formula,                    ; concept:<abstraction>(<slots>)
    rhs:               ir-formula                     ; the per-language operation-layer term it expands to
  },
  role:                "abstraction-realization",     ; NEW; elects it into the abstraction-realizer
  direction:           "left-to-right",               ; the lhs is the abstraction, the rhs is the lang term
  target_lang:         tstr,                           ; NEW; LSP language id ("c11", "python", "rust", ...)
  loss_record:         loss-record,                    ; NEW; the multidimensional record, §2.4
  discharge_receipt:   cid,                             ; NEW; CID of a MorphismDischargeReceipt; §2.5
  effects:             [],                              ; the equation itself has none; the rhs term carries effects
  ? refines:           cid
}
```

Multiple realizations per `(abstraction, lang)` are allowed and expected. C's `concept:dynamic-dispatch` has a `→ vtable-struct` realization for the static-layout case (`pre`: the dispatch table is fixed at compile or link time) and a `→ runtime-hashtable` realization for the dynamically-extended-table case (`pre`: the table may gain entries at runtime), with different loss records. The realizer (§5.3) picks among them by the source program's needs plus the per-dimension `loss-budget`. That is the `TransportGapMemento`-with-`resolution_options` structure from the 2026-05-14 spec, at this tier: a "this abstraction's realization in language L over your budget" gap names the budget the realizer would need.

### §2.3 The contract formula

The contract is the abstraction's `wp_rule` in the 2026-05-13 language: a predicate-transformer rule, an `ir-formula` over the slot variables plus the reserved meta-variables `Q` and `wp_<slot>`. For `concept:dynamic-dispatch` with slots `receiver`, `method_name`, `args` (`args` variadic), the clause says: the call's result and effect equal those of `lookup(runtime_type(receiver), method_name)` applied to `[receiver, ...args]`. As a transformer:

```
wp_rule(Q)  ≡  let m = lookup(runtime_type(receiver), method_name) in
               defined(m)  ∧  wp_call(m, [receiver, ...args])(Q)
```

This needs two function symbols the op-layer `wp`-formula vocabulary does not have yet: `runtime_type(·)` (the runtime type of a value, an opaque sort `Type`) and `lookup(Type, Name)` (the resolution function, returning a `Method` or undefined). They are abstraction-level interpreted symbols, axiomatized in a small theory the discharge (§2.5) and the realization desugarings share. **Open question, flagged for the proposal:** the 2026-05-13 `wp`-formula language was sized for op-layer control-flow. Abstraction-level contracts quantify over *runtime types* and *dispatch tables*, which is one quantifier-rank higher. The likely answer is that the formula language stays the same (it already has `∀`) and what is new is the *theory*: a `dispatch-theory` SortMemento bundle (`Type`, `Name`, `Method`, `lookup`, `runtime_type`, `defined`) with its axioms, the same way `LoopInvariantMemento` plugs an invariant theory into `concept:while`. If quantifier-rank turns out to be a real obstacle for some abstraction, that abstraction's contract degrades to a weaker characterization plus a larger `structural_divergence`, never to "no contract." That is a thing to settle empirically, not by fiat.

### §2.4 The `loss_record` (and the one dimension this tier adds)

The `loss-record` of the 2026-05-14 spec, reused, one tier up, plus one new dimension. #616's four dimensions carry through unchanged:

- **`value_divergence`** (#616): `∅` for a well-typed use. The abstraction's *result value* on well-typed inputs is preserved by every realization. A realization that changed the value would not be a realization, it would be a different program (*Supra omnia, rectum*). Non-`∅` only for ill-typed uses, which the lift refuses upstream anyway.
- **`ub_introduction`** (#616): non-empty when a realization replaces defined behaviour with undefined behaviour. At the abstraction tier this is real for the C-side realizations: `concept:reference → c11:raw-pointer` introduces use-after-free / aliasing UB on the states where the C-origin (or transported-into-C) program's pointer outlives its referent; `concept:iterator → c11:hand-rolled-cursor` introduces off-by-one read UB on the states where the cursor is advanced past the length. The §3 and §4 examples below work cases where `structural_divergence` is the load-bearing dimension and are silent where `ub_introduction` is empty; the non-empty cases (`concept:reference → C`, `concept:iterator → C`, and the closure-env-pointer case) are recorded in the catalog spec (§8).
- **`domain_narrowing`** (#616): non-empty exactly when the target's realization cannot honour a capability the abstraction (or, on the way out, a *richer* realization) exposes. Example: Python's dict-protocol realization of dispatch exposes runtime mutability of the dispatch table (`setattr`, `__class__` reassignment); a C vtable-struct realization cannot honour that; so transporting a Python dispatch site to C narrows the domain to "program states where the dispatched-upon object's class and method set are not mutated after construction." Recorded as the predicate carving out the kept states.
- **`effect_divergence`** (#616): non-empty when the encoding needs machinery the abstraction does not promise: heap allocation for a closure environment that outlives its stack frame, heap for a runtime hashtable, an exception-table or `setjmp` buffer for an exception realization in a language without native exceptions.

This spec **proposes adding a fifth dimension, `structural_divergence`, specific to the abstraction tier**: the realized code's surface form does not resemble the abstraction. An open-coded pointer chain is not a dispatch primitive, a name-lookup protocol is not a primitive, a defunctionalized closure is not a lambda. This dimension records *how far* the surface form of the realization is from the abstraction; for an abstraction realization it is *always non-empty*, because if it were empty the hub node would be an op, not an abstraction. This dimension has **no analog at the op tier** (`python:add → c11:add` is structurally identical; op-to-op transports preserve op structure by construction), which is exactly why #616 did not need it: it becomes load-bearing at the moment the hub node is a *pattern* rather than an *op*. Adding it to `loss-record` is a successor mint of the #616 schema per LSP §4.4 (`refines = <2026-05-14 loss-record CID>`), not a fork: every #616 `loss-record` reads as a `structural_divergence: ∅` instance of the extended one, the same way every empty-loss bridge reads as the exact case in #616's own scheme. The `coarsen_d` rewrite (#616 §3) gains a `structural_divergence` clause: coarsening in that dimension is the identity on the `wp` formula and a non-identity on the *re-sugar/presentation* obligation (the realized form is admitted even though it does not look like the abstraction), which is why `structural_divergence` never weakens the *behaviour* claim, only the *form* claim.

`loss_record`s compose, per dimension, through a program's dataflow, exactly as in the 2026-05-14 spec (the `structural_divergence` dimension composes too: a program with three abstraction realizations has the union-modulo-dataflow of their structural divergences as its surface-form divergence record). The composed record of `realize ∘ transport ∘ re-recognize` (§5.2) is the loss profile of the abstraction-transported program.

### §2.5 The discharge

A `RealizationDesugaringMemento` discharges iff its `post` equation is `wp`-preserving modulo `loss_record`: `wp(lhs, Q) ⊑ wp(rhs, Q)` for all `Q`, where `⊑` is the refinement-up-to-`loss_record` ordering (the abstraction's `wp` is at least as strong as the realization's `wp` on the states the realization keeps, i.e. on the complement of `domain_narrowing`, with `effect_divergence` accounted as added-but-declared effects). This is a 2026-05-13 §3 refinement check: structural match first (the realization term's `wp_rule`, unfolded through the op-layer `wp`s of its constituent ops, equals the abstraction's `wp_rule` up to the loss record's accounting), then a Z3 check over the dispatch-theory axioms, then a gap. A discharged realization produces a `MorphismDischargeReceipt`; its CID is the `discharge_receipt` field. A failed discharge is a gap, not a warning: the realization is not minted, and the `(abstraction, lang)` cell is recorded in `transport-gaps.md` (now the per-op-pair *and* per-abstraction-pair projection, §5.4) with the structural reason and actual-vs-expected. If *no* realization for `(abstraction, lang)` discharges, then by the §0.1 admissibility criterion that is a signal: either the realization was authored wrong, or `lang` is not actually a target.

## §3: `concept:dynamic-dispatch`, the worked example

### §3.0 Why {C, Java, Python} is the validation trio

The spec works `concept:dynamic-dispatch` (and the other abstraction-concepts in §4) rigorously for three languages, C and Java and Python, and sketches the rest in a sentence or two each. That is a strategy, not a corner cut, and it is the answer to "why these realizations and not an exhaustive ten-language matrix in the spec."

C, Java, and Python *bracket the design space*:

- **C is the unforgiving end.** No high-level abstraction is native. Everything is an encoding-with-structural-loss: `concept:dynamic-dispatch → vtable-struct + deref + fn-ptr-call`, `concept:closure → defunctionalization + explicit-env-struct`, `concept:exception → result-passing`, even `concept:Int → fixed-width-with-UB-overflow`. If a realization-desugaring discharges into C, it discharges. C exercises `structural_divergence` and `effect_divergence` hardest, because C is where every abstraction becomes a pile of ops.
- **Python is the loose end.** The abstractions are first-class, so the loss going *in* is small. The loss going *out* is the interesting one: `domain_narrowing`. `setattr` and monkey-patching and unbounded `int` are *exposed capabilities*, and they are exactly what you restrict when you realize a Python program into anything stricter. Python is also the hardest *lift* target: syntax-only, so the re-recognition pass (§6.2) and the partial-morphism-precondition problem bite here first. Python exercises `domain_narrowing` and the lift-side conservatism hardest.
- **Java is the middle.** GC and exceptions and OO are native, but it is static-typed, its generics are type-erased, and it had no first-class lambda syntax before 8 (the closure realization pre-8 is an anonymous inner class, which has been there since 1.1). Java exercises the "native but constrained" cases: dispatch is near-first-class but `final` blocks it, generics exist but erase, closures have lambda syntax now but the older realization is still an anonymous class.

Validating the concept library, the realization-desugarings, and the loss records across that trio validates the design, period. TypeScript, Ruby, PHP, Go, Rust, Zig, Swift, C# all fall *within* the C-to-Python span: each is a mechanical interpolation, a column-add, no new concepts. Rust adds the over-*constrain* wrinkle (borrow obligations, object-safety) but that is a `domain_narrowing` going *into* Rust, a known shape, not a new dimension. So the order of operations is: get the trio solid, then mechanically fill the other seven. The sketches in §3 for Rust/Go/JS/C# and the sentence-each for TS/Zig/Ruby/PHP are the interpolation map, not the validation set.

### §3.1 Identity and contract

**Identity.** `operator: "concept:dynamic-dispatch"`, `tier: "abstraction"`, slots `[receiver, method_name, args (variadic)]`, `formal_sorts: [Value, Name, [Value]]`, `result_sort: Value`.

**Contract.** As §2.3:

```
wp_rule(Q)  ≡  let m = lookup(runtime_type(receiver), method_name) in
               defined(m)  ∧  wp_call(m, [receiver, ...args])(Q)
```

In prose (`contract_note`): "the call's result and effect equal those of the method that resolves from the receiver's *runtime* type for `method_name`, applied to the receiver and the arguments; if no such method resolves the behaviour is undefined (the realizations narrow this differently, see each `domain_narrowing`)."

### §3.2 The validation trio, rigorously

**C, the unforgiving realization (`concept:dynamic-dispatch → c11:vtable-indirection`, static-layout case).** `rhs = concept:call(concept:member(concept:deref(concept:member(receiver, "vtbl")), method_name), [receiver, ...args])`, read: dereference the receiver's `vtbl` field to get the vtable, index it by `method_name` (a compile/link-time-fixed offset), call that function pointer with `receiver` first. `pre`: the dispatch table for `runtime_type(receiver)` is fixed at compile or link time (no runtime table mutation). `loss_record`: `structural_divergence = {"open-coded pointer chain, not a dispatch primitive"; "the offset is fixed, the abstraction's lookup is by name"}`; `domain_narrowing = ∅` for the static case, since the C realization over-delivers a static-layout guarantee (the table is immutable, the offsets are constant), which is harmless on the way in and is *more* than the abstraction promises; `effect_divergence = ∅`; `value_divergence = ∅`. Discharge: unfold the `rhs` through the op-layer `wp`s of `concept:deref`, `concept:member`, `concept:call`; the composed transformer says "apply the function found at the fixed `vtbl[method_name]` slot to `[receiver, ...args]`"; under the `pre` (the table is the fixed one for `runtime_type(receiver)`) that equals `wp_call(lookup(runtime_type(receiver), method_name), [receiver, ...args])`, which is the abstraction's `wp_rule` after the dispatch-theory axiom "the vtable for a type *is* its `lookup` table." Structural match modulo that axiom; no Z3 call. The receipt records the axiom set used.

**C, the alternative (`concept:dynamic-dispatch → c11:runtime-hashtable`, dynamic-table case).** `rhs = concept:call(concept:hashtable-lookup(concept:member(receiver, "dispatch_tbl"), method_name), [receiver, ...args])`, a runtime hashtable keyed by method name, allocated and possibly extended at runtime. `pre`: the dispatch table may gain entries at runtime. `loss_record`: `structural_divergence` heavier than the vtable case (a hash computation, a probe loop); `domain_narrowing = ∅` (it *can* honour runtime extension); `effect_divergence = {heap allocation for the table}`; `value_divergence = ∅`. Discharge: same shape, with `concept:hashtable-lookup`'s `wp` axiomatized as "returns the entry for the key if present, undefined otherwise," which matches `lookup` directly; the heap effect is *declared*, so it composes under the 2026-05-13 effect-⊆ direction (the abstraction's effect set is a subset of the realization's). The realizer picks this one over the vtable one when the source program mutates the dispatch table, or when the budget forbids the `domain_narrowing` the vtable realization would otherwise force on a source that needs mutation.

**Java, the native-but-constrained realization (`concept:dynamic-dispatch → jvm:virtual-method`).** `rhs = concept:call(concept:vtable-method(receiver, method_name), [receiver, ...args])` over the JVM object header's per-class method table (a `virtual` method or an interface method via the itable). `pre`: `method_name` resolves to a non-`final`, non-`static`, non-`private` method on the receiver's class hierarchy (i.e. a virtually dispatched one). `loss_record`: `structural_divergence` *low* (the JVM has dispatch near-first-class, the realization is `invokevirtual` / `invokeinterface`, barely a desugaring); `domain_narrowing = {dispatch over a method that is `final` or `sealed` in the target hierarchy, or `static`, or `private`}` (those are statically bound on the JVM; a source whose dispatch site needs full openness over a method the Java target sealed narrows there, and the lift must establish the method is not so sealed or refuse); `effect_divergence = ∅` (the JVM's heap and class-loading are not effects the realization adds beyond what the abstraction allows); `value_divergence = ∅`. Discharge: `concept:vtable-method`'s `wp` is "apply the most-derived override of `method_name` for `runtime_type(receiver)`," which is `lookup(runtime_type(receiver), method_name)` once the dispatch-theory axiom "the JVM's most-derived-override resolution *is* `lookup`" is in scope; structural match modulo that axiom under the `pre`. Class loading is a side effect the abstraction's `lookup` is silent on; treated as a declared environment effect, composes the effect-⊆ way.

**Python, the loose realization (`concept:dynamic-dispatch → python:mro-dict-lookup`).** `rhs = concept:call(concept:bound-method(concept:dict-lookup-over-mro(concept:type-of(receiver), method_name), receiver), [receiver, ...args])`, walk `type(receiver).__mro__`, look `method_name` up in each class `__dict__` in order, bind the found function to the receiver, call it. (`concept:dict-lookup-over-mro` and `concept:bound-method` are op-layer mints this spec calls for, §8; `concept:type-of` is `runtime_type` reified as an op.) `pre`: true (Python's realization is the most permissive). `loss_record`: `structural_divergence = {"name-lookup protocol over a list of dicts, not a primitive"}`; `domain_narrowing`, and here is the subtlety: Python's realization *over-delivers* introspectability and mutability of the dispatch table. That is harmless on the way *in* (a C-origin or Java-origin program transported to Python just does not use the extra capability). It is exactly that exposed capability that creates a `domain_narrowing` going *out* to a language whose realization cannot honour it. Characterize it on the *outbound* leg, not on the node: `domain_narrowing = {program states reachable via setattr / delattr / __class__-reassignment on a dispatched-upon object or its type}`. That is why Python → C and Python → Java dispatch are *partial morphisms* (`PartialMorphismMemento` in the 2026-05-14 sense): valid under the side condition "no runtime dispatch-table mutation on the dispatched objects," which the lifter must *establish* (a no-mutation invariant on the relevant types), else refuse, or fall back to the `runtime-hashtable` C realization which can honour it. `effect_divergence = ∅` (Python's heap is implicit, the abstraction does not promise stack discipline; `__getattr__` / descriptor protocol side effects, if the source uses them, push this non-empty and are recorded then); `value_divergence = ∅`. Discharge: `concept:dict-lookup-over-mro`'s `wp` is "the first `method_name` entry found scanning `__mro__` in order," which equals `lookup(runtime_type(receiver), method_name)` once the dispatch-theory axiom "Python's MRO scan *is* `lookup`" is in scope, plus the descriptor-protocol axioms if the source touches them; structural match modulo those axioms; if descriptors are involved the residual goes to Z3.

The trio also brackets the *lift* side: lifting C dispatch is recognizing a fixed pointer chain (mechanical given the realization-desugaring read backwards); lifting Java dispatch is recognizing `invokevirtual`/`invokeinterface` (trivial, it is a single bytecode); lifting Python dispatch is the hard one, recognizing `obj.method(args)` *and proving the relevant types are not monkey-patched* before you may call it `concept:dynamic-dispatch` rather than three unrelated attribute/call ops (the conservative-recognition discipline of §6.2 bites here).

### §3.3 The interpolates, sketched

These fall within the C-to-Python span; each is a column-add, recorded here as a sentence or two, with the full realization-desugaring deferred to the catalog spec (§8).

- **Rust (`concept:dynamic-dispatch → rust:trait-object`).** `rhs` a call through a `dyn Trait` fat pointer (data pointer plus vtable pointer). `structural_divergence` low (near-first-class). The wrinkle: Rust *over-constrains*, `domain_narrowing = {dispatch sites whose method set is not object-safe: generic methods, methods returning Self, associated-type returns}`; translating *to* Rust narrows there, translating *from* Rust is free. `effect_divergence = ∅`, `value_divergence = ∅`.
- **Go (`concept:dynamic-dispatch → go:interface-value`).** `rhs` a call through an interface value's itab. `structural_divergence` low. `domain_narrowing = {dispatch over methods not in any declarable interface}` (narrower than Rust's object-safety, same shape). `effect_divergence = ∅` except the receiver-not-yet-an-interface conversion case, which allocates (a sub-realization). `value_divergence = ∅`.
- **JavaScript (`concept:dynamic-dispatch → js:prototype-chain`).** `rhs` a `concept:proto-walk(receiver, method_name)` then call: walk `receiver.__proto__` links until `method_name` is an own property. `structural_divergence` = "prototype-link walk, not a primitive; the chain is prototype objects, not MRO dicts, so the resolution order differs from Python's." `domain_narrowing` on the *outbound* leg, same shape as Python: `{states reachable via __proto__ reassignment, property deletion, shadowing}`. `effect_divergence = ∅`, `value_divergence = ∅`.
- **C# (`concept:dynamic-dispatch → clr:virtual-method`).** Like Java's `jvm:virtual-method`: `structural_divergence` low, `domain_narrowing = {dispatch over a `sealed` or `static` method}`. (`dynamic` and DLR call-sites are a separate, lossier sub-realization closer to the JS prototype story.) `effect_divergence = ∅`, `value_divergence = ∅`.
- **TypeScript / Swift / Ruby / PHP / Zig.** TS desugars to the JS prototype realization (TS dispatch *is* JS dispatch at runtime; the static types are erased, recorded under `structural_divergence`). Swift has a protocol-witness-table realization, near-first-class, with a `domain_narrowing` for non-`@objc` dynamic-replacement cases. Ruby is the Python story with its own method-resolution order and `method_missing` (the descriptor-protocol analogue). PHP is the Java story with magic `__call`. Zig is the C story plus comptime: tagged-union-and-switch or fn-ptr-table, `structural_divergence` heavy, `domain_narrowing = ∅` for the comptime-fixed case. Each is mechanical fill once the trio is solid.

What makes `domain_narrowing` non-empty in every case is the capability the *richer* side exposes that the *weaker* realization cannot honour: runtime table mutation (Python/JS/Ruby expose it, the vtable family does not), open extensibility of a method (`final`/`sealed` on the JVM/CLR/PHP, object-safety in Rust, interface-membership in Go each restrict it differently). The vtable family `(C-static, Java, C#, Go, Rust, Swift, Zig-comptime)` shares structure (a fixed table indirection); the dict/prototype family `(Python, JS, TS, Ruby)` shares structure (a name-lookup walk); the C `runtime-hashtable` realization is the bridge case (a vtable-family language reaching for the dict-family capability by hand, paying `effect_divergence`).

## §4: The other core abstraction-concepts

One paragraph each. Less worked than dispatch; the full realization specs land in the follow-up catalog spec (§8). Each paragraph leads with the {C, Java, Python} bracket (the validation trio of §3.0), then names the interpolates; that is the same C-unforgiving / Java-mid / Python-loose span, one abstraction at a time.

**`concept:closure`**: a function plus its captured environment. Slots `[body, captured_vars, params]`. Native in Python, Rust, JS, Swift, modern Java, modern C#: `structural_divergence` near zero, the realization is the language's lambda. In C: defunctionalization plus an explicit environment struct passed as an extra parameter (`rhs` a `concept:call` over a function pointer with an env-struct argument); `structural_divergence` heavy; `effect_divergence = {heap allocation}` exactly when the environment outlives the stack frame, `∅` when it does not (escape analysis is the side condition; two sub-realizations, stack-env and heap-env). In old-Java / pre-generics Go: an interface with captured fields, or an anonymous class; `structural_divergence` heavy (a one-method object standing in for a function); `effect_divergence = {heap}` for the object. Contract: applying the closure equals applying `body` with `params` bound to the call arguments and the captured names bound to the values they had at capture time. The 2026-05-05 `ClosureBindingMemento` is the op-layer/contract-level companion; this node is the cross-language abstraction it realizes.

**`concept:exception`**: non-local transfer of control carrying a payload, unwinding to the nearest matching handler. Slots `[try_body, handlers, throw_payload]` (and `concept:throw` is the op-layer node it raises through). Native in Python, Java, JS, C#, Ruby: `structural_divergence` low. In Rust: `Result<T, E>` plus `?`-propagation (`rhs` a `concept:conditional` on the result tag at every fallible call site, threading the error out); `structural_divergence` heavy (the control flow becomes explicit data flow); `domain_narrowing = ∅` mostly (any unwind is expressible as error-return); `effect_divergence` minor (none, actually: it is *less* effectful than native unwinding, which composes the other way per the 2026-05-13 effect-⊆ relaxation). In Go: `panic`/`recover`, or the error-return-and-check-at-every-call idiom; same shape as Rust. In C: result-passing, or `setjmp`/`longjmp` (the latter `effect_divergence = {a jump buffer}`). The 2026-05-05 `TryBranchMemento` is the op-layer companion. Contract: a `throw` inside `try_body` transfers control to the first `handler` matching the payload's type, with the payload bound; absent a match it propagates to the enclosing `concept:exception`.

**`concept:reference` / `concept:aliasing`**: a name bound to a mutable cell that other names may also be bound to, so a write through one is observable through another. Slots `[referent]`. Raw pointer in C (`rhs` `concept:addr` / `concept:deref`); object identity in Python, Java, JS (everything-is-a-reference, the realization is the identity map, `structural_divergence` near zero). Rust is the interesting one: `&` / `&mut` *with the borrow obligations*, and that means Rust's realization *adds* obligations the abstraction does not promise (no aliasing-while-mutating, lifetimes bounded by the referent's scope). So Rust *over-constrains*. Translating *to* Rust is a `domain_narrowing` (only borrow-checkable aliasing patterns survive: no aliased mutation, no self-referential structures without `Pin`, no upward funarg of a `&mut`); translating *from* Rust is free (Rust's `&`/`&mut` use is a special case of unrestricted aliasing, the obligations just stop being checked). This is the *dual* of the Python-dispatch case: Python's realization is too *permissive* (exposes a capability), so going out of Python narrows; Rust's realization is too *restrictive* (imposes an obligation), so going *into* Rust narrows. And the obligations have to be *discharged* by the lift if they survive a round-trip: a Rust `&mut` realization of `concept:reference` carries a borrow obligation that a C-origin program transported through it must satisfy. Does it? Characterize: the C-origin program's aliasing must already be uniqueness-checkable at the point the `&mut` is taken, which is *not* automatic; so C → (concept:reference) → Rust is a `PartialMorphismMemento` whose side condition is "the C aliasing pattern is borrow-checkable," and the lift must establish it (a uniqueness/non-escape invariant) or refuse. The 2026-05-05 `OutlivesKernelAxioms` and the `PinInvariantMemento` are the op-layer machinery the borrow obligations discharge through.

**`concept:iterator`**: a uniform protocol for stepping through a sequence: get an iterator, ask it for the next element, repeat until exhausted. Slots `[source, step_var, body]` for the loop form; `[iterator_handle]` for the bare protocol. This is the `concept:foreach` thing the 2026-05-14 work *deferred* (the `fa031012` commit, "foreach has no common iterator protocol across the 10 languages"), now framed properly: there is no common iterator *op*, but there *is* a common iterator *abstraction*, and `concept:foreach` (an op-layer surface form) desugars to `concept:iterator-protocol-loop` (this abstraction node) which has these per-language realizations: Python `__iter__`/`__next__` with `StopIteration`; Java `Iterable`/`Iterator` with `hasNext`/`next`; Go `range` (a built-in stepping form) or the Go 1.23 range-over-func protocol; C++ `begin`/`end` with `!=` and `++`; Rust `IntoIterator`/`Iterator::next` returning `Option`; C a hand-rolled cursor (an index and a length, or a `next` function pointer). `structural_divergence` ranges from low (Python, Rust, where iteration *is* this protocol) to high (C, where it is a hand-rolled loop). `domain_narrowing` is mostly `∅` for finite sequences; for infinite/lazy sequences a language without lazy iteration (C, naive) narrows to "the iteration terminates." Contract: the loop body runs once per element of `source` in `source`'s order, with `step_var` bound to that element, and stops when `source` is exhausted, with no observation of an element out of order or after exhaustion.

**`concept:generic-instantiation`**: a type-parametric definition specialized at a point of use. Slots `[parametric_def, type_args]`. Monomorphization in Rust and C++ (`rhs` a fresh copy of `parametric_def` with `type_args` substituted; `structural_divergence` = "the def is duplicated per instantiation, code size blows up"). Type-erasure plus casts in Java (`rhs` the def with the type parameters erased to their bounds and casts inserted at use sites; `structural_divergence` = "casts inserted, the type info is gone at runtime"). Duck-typing-with-no-instantiation in Python (the "generic" is *implicit*: there is no instantiation site, the def just works on whatever is passed; the realization is *the absence of one*). That last one is the interesting case: a Rust generic transported to Python *loses the monomorphization-perf and the type-checking-at-instantiation*, but the *behaviour* on well-typed inputs is preserved. So `value_divergence = ∅`, `domain_narrowing = ∅` (well-typed inputs all work; ill-typed ones fail at run time in Python instead of compile time in Rust, which is a *diagnostic-timing* loss, recorded under `structural_divergence`, not a behaviour loss), `effect_divergence = ∅`. Contract: an instantiation of `parametric_def` at `type_args` behaves as `parametric_def` with the type parameters bound to `type_args`, on inputs that satisfy the parameters' bounds.

## §5: The mechanics: the operation layer, one tier up

### §5.1 Composition through the hub: still M + N

Every language realizes every abstraction *to* its own operation-layer term (M realization-desugarings, the rows in §3 and §4, M = number of languages times number of abstractions, but with heavy structure-sharing inside the vtable-family and dict-family clusters so the *distinct* realization shapes are far fewer than M). The operation-layer machinery already transports those terms (the N op-layer morphisms, unchanged). So the abstraction layer is M + N, not M × N: every language realizes every abstraction to its op-term, then the existing op-layer hub transports the op-term. No M × N table of "Python dispatch to C dispatch" entries. The diff between any two languages' dispatch realizations is *computed* from the two loss records plus the op-layer transport, never tabulated. This is the org-frame thesis (`provekit migrate` is a loss-composition engine; M × N is impossible, M + N is the topology, composable loss-records make every M × N loss-diff computable) holding at the abstraction tier.

### §5.2 Cross-language abstraction transport, end to end

To transport a program from source language S to target language T where the program uses abstraction `A`:

1. **Re-recognize** on the lift side: the lifted S program contains a `deref + member + call` chain (or a `dict-lookup + call` chain, or a `Result`-threading chain, ...); recognize that this chain *is* `concept:A`, i.e. lift it from the op-layer term up to the abstraction node. This is `re-recognize(tgt-op-term → tgt-abstraction-realization)` run in reverse on S: match S's realization-desugaring for `A` backwards. (§6 names why this is the hard part.)
2. **Realize** into T: desugar `concept:A` forward through T's realization-desugaring, `realize(abstraction → tgt-op-term)`, picking among T's realizations by the source program's needs and the `loss-budget` (§5.3).
3. **Transport** the resulting op-layer term via the existing op-layer morphisms, `transport(src-op-term → tgt-op-term)` (this step is unchanged from PTP §2; it operates on `concept:core`).

The loss of the whole is the composition, per dimension, through the program's dataflow, of three loss records: the re-recognition loss (usually `∅` if the recognition was conservative, see §6; a *lossy recognition*, recognizing a chain as `A` when it is *almost* `A`, is exactly a `LossyMorphismMemento` and is forbidden in the conservative regime), the source realization's loss record (read *backwards*: the capabilities S's realization exposed become the `domain_narrowing` if T's chosen realization cannot honour them), and T's realization's loss record. The fidelity-domain of the transported program (the 2026-05-12 PTP notion) now includes abstraction-layer losses too.

### §5.3 The realization picker

When T has more than one realization for `A` (C's vtable vs hashtable; Go's allocating vs non-allocating interface conversion; the closure stack-env vs heap-env), the realizer picks the one that (a) satisfies its `pre` given the source program's facts, and (b) keeps the composed loss record within the per-dimension `loss-budget`. If no realization does both, that is a `TransportGapMemento`-with-`resolution_options` at this tier: the gap names which budget dimension the realizer would need raised (e.g. "C dispatch realization needs either `domain_narrowing` for table mutation lifted, or `effect_divergence` for heap lifted; the source mutates the table, so the vtable realization needs the first, the hashtable realization needs the second; pick one"). The constructive over-budget refusal (2026-05-14) applies: the substrate refuses with the precise budget-extension request, it does not silently pick the lossier realization.

### §5.4 `transport-gaps.md`

The flat `transport-gaps.md` file is, after this spec, the static per-op-pair *and per-abstraction-pair* projection of the dynamic per-program loss composition: each row is "(S-realization-of-A, T-realization-of-A): composed loss record, or `[no T realization discharged]`." It is the same role it plays for op-pairs today, extended a tier. A row is a fact about the *worst case* over all programs using `A`; a particular program may compose to less loss (if its dataflow does not reach the narrowed states).

## §6: What it costs, what it is worth, the hard parts

### §6.1 The work

- The `ConceptAbstractionMemento` schema and the `RealizationDesugaringMemento` schema (CDDL above), plus the canonicalizer recognizing the new `role` and the new `tier`.
- The abstraction-layer hub catalog: `concept:dynamic-dispatch` plus the ~6 core ones in §4 (`closure`, `exception`, `reference`/`aliasing`, `iterator`, `generic-instantiation`), each with its contract formula and slot structure.
- The follow-up op-layer hub mints the realizations need as targets: `concept:dict-lookup`, `concept:dict-lookup-over-mro`, `concept:hashtable-lookup`, `concept:bound-method`, `concept:proto-walk`, `concept:vtable-method`, `concept:itab-method`, `concept:type-of` (each a small op-layer node in the §1.1 sense, minted the same way `concept:deref` was).
- Per-language realization-desugarings: roughly 6 to 8 languages times roughly 6 abstractions, but the vtable-family realizations share a shape, the dict/prototype-family realizations share a shape, the native-family realizations are near-identity, so the *distinct authored* desugarings are perhaps 15 to 25, not 48.
- The re-recognition lift pass (§6.2).
- The realizer's realization-picker (§5.3): a small extension to the 2026-05-10 Obligation Realizer v2, choosing among the realizations registered for `(A, T)` by `pre` plus budget.
- Loss-composition through the three stages of §5.2: an extension of the 2026-05-14 composition machinery to chain `re-recognize ∘ transport ∘ realize`.

### §6.2 The hard parts, named honestly

- **Re-recognition.** Lifting an S program means recognizing that a `deref + member + call` chain in C *is* `concept:dynamic-dispatch`, versus three unrelated ops that happen to look like it. This is a pattern-matching / idiom-recognition pass, in the same family as the Pattern-Predicate Protocol (PPP, 2026-05-09) work. It is **undecidable in general** (recognizing an arbitrary encoding of an abstraction is recognizing an arbitrary program property), so it is **heuristic in practice**, and crucially, a mis-recognition is a *correctness* hazard, not a quality hazard: if the lifter says "this chain is dispatch" and it is not, every cross-language fact transported through that node is a lie. So the recognition pass MUST be **conservative**: recognize a chain as `concept:A` *only when it can prove the chain is the abstraction*, i.e. only when the chain matches S's realization-desugaring for `A` up to a discharged equality. When it cannot prove it, the chain stays as op-layer ops and transports as such (lossier in the *abstraction* sense, the target gets an open-coded vtable chain, not a `dyn Trait`, but never *wrong*). This is the 2026-05-14 "conservative over lossy" discipline applied to recognition: a `LossyMorphismMemento` on the recognition step (recognizing an *almost*-dispatch chain as dispatch) is forbidden; the recognition is exact or it does not happen.
- **The abstraction-contract formula language.** It is the 2026-05-13 `wp`-formula language, but abstraction-level contracts quantify over runtime types and dispatch tables (§2.3). The likely resolution is "same formula language, new shared theory" (the `dispatch-theory` bundle), the same pattern `LoopInvariantMemento` uses for `concept:while`. Whether the formula language is *rich enough* for every abstraction's contract is an open empirical question, flagged here, with the fallback being a weaker contract plus a larger `structural_divergence`, never "no contract."
- **The realizations that over-constrain.** Rust's borrow obligations, Rust's trait object-safety, Java's `final`, Go's interface-membership. These are `domain_narrowing` *into* the language. But the obligations also have to be *discharged by the lift* if they survive a round-trip: a Rust `&mut` realization of `concept:reference` carries a borrow obligation that a C-origin program transported through it must satisfy, and that obligation is *not* automatic, so the relevant `PartialMorphismMemento`'s side condition ("the C aliasing is borrow-checkable") is a real obligation the lift establishes or refuses on. Characterized in §3 (Rust dispatch object-safety) and §4 (`concept:reference`). The dual of this, realizations that over-*permit* (Python's mutable dispatch table, JS's `__proto__` reassignment), is the `domain_narrowing` on the *outbound* leg, also characterized.

### §6.3 The payoff

Cross-language transport works for *real* programs (the ones that are full of dispatch, closures, exceptions, iterators, generics, not just `1 + 1`), with the loss characterized at the abstraction level, per dimension, per program. The "no permanent gaps, only loss-profiles" thesis (2026-05-14) holds at the abstraction layer too: there is no abstraction that "cannot go to language L," there is a realization of it in L with a loss record, and if there genuinely is no discharging realization that is a signal that L is not a target (§0.1), not a gap to live with. And it is the operational realization of "the canonical form is the concept, every language is a lossy projection": the hub node `concept:dynamic-dispatch` is the canonical thing, C's vtable and Python's dict are *both* projections of it, and the diff between them is computed, not tabulated.

## §7: Relationship to the rest

- **Transport Gap and Partial Morphism Protocol (2026-05-14, PR #616, landed).** The `TransportGapMemento` / `PartialMorphismMemento` / `LossyMorphismMemento` types, the multidimensional `loss-record`, the `loss-budget` (per-dimension tolerance, the Pareto-frontier candidate ordering), the `coarsen_d` rewrite, the constructive over-budget refusal, the per-program `fidelity-domain` record: the abstraction layer reuses all of it, one tier up. A realization's `loss_record` is a `loss-record` (extended with the `structural_divergence` dimension of §2.4, a successor mint of #616's schema per LSP §4.4, not a fork). A `(Python, C)` dispatch transport is a `PartialMorphismMemento`. A realization-picker's "no realization within budget" is a `TransportGapMemento` with `resolution_options`. A recognition that is not exact would be a `LossyMorphismMemento`, which is why it is forbidden. The only addition this spec makes to #616 is the one dimension; everything else is consumed verbatim, and the dimension carries through #616's whole apparatus (the `coarsen_d` clause, the budget component, the candidate ordering, the fidelity-domain composition) by being a `loss-record` dimension like the other four.
- **`wp` as Formula (2026-05-13, PR #613).** An abstraction-concept's contract is a `wp_rule` in that formula language; a realization-desugaring's loss record's `domain_narrowing` is a formula in that language (the predicate carving out the kept states); the discharge of a realization-desugaring is a 2026-05-13 §3 refinement check (`wp(lhs) ⊑ wp(rhs)` modulo the loss record), structural-match-first then Z3-over-the-dispatch-theory then a gap.
- **Desugaring and the O(N) Core Compression (2026-05-11, PR #601).** A realization *is* a desugaring: `concept:abstraction → <lang>-operation-layer-term`, a `DesugaringEquationMemento` elected into the `abstraction-realization` role, oriented left to right, carrying the §1.2 `wp`-preservation obligation (here, modulo the loss record). The desugaring layer is the machinery; this spec adds a *role* and three fields.
- **Obligation Realizer Protocol v2 (2026-05-10, PR #67-family).** That protocol is the *control-flow* realizer: `concept:goto → state-machine`, the encodings of irreducible control. This spec is the *abstraction* realizer: `concept:dynamic-dispatch → vtable-struct | dict | trait-object`, the encodings of patterns/protocols/capabilities. Same realizer library, this tier; the realization-picker (§5.3) is a small extension to the v2 realizer's selection logic.
- **Pattern-Predicate Protocol (2026-05-09).** The re-recognition pass on the lift side (§6.2) is pattern-matching: recognizing that an op-layer term *is* an abstraction. That is in the PPP family: a realization-desugaring read backwards is a pattern, the recognition is matching that pattern with a proof obligation attached (recognize only when you can *prove* it).
- **Paper 13 (After Grammars).** A realization is a morphism `concept:abstraction → <lang>-subalgebra` (the subalgebra generated by the ops in the realization term), in the Lemma 4 sense: the discharge of the realization-desugaring is the cross-language transferability proof for facts about that abstraction, one rung above the op-morphism Lemma 4 already covers.
- **Paper 07 (the structural-elimination theorem).** An abstraction's contract `wp` is its *structural eliminator*: the rule that says how to reason about a use of the abstraction by reducing it to reasoning about its meaning (`lookup(runtime_type(receiver), method_name)` applied to the args), the same way `concept:while`'s `wp_rule` is the eliminator for loops via the invariant. An abstraction with no eliminator is an abstraction you cannot reason through; this spec requires every abstraction node to carry one (possibly weak, never absent).
- **The M + N hub + loss-composition org-frame.** The abstraction layer is that frame, one tier up: same M + N topology (every language realizes every abstraction to its op-term, the op-layer hub transports the op-term), same composable loss records (the three-stage composition of §5.2), every M × N abstraction-transport loss-diff computable without an M × N table.

### §7.1 The design criterion, as a first-class statement

A target language must realize *every* core concept, operation-layer and abstraction-layer, possibly lossy, always characterized. A language that genuinely cannot is not a target. That is a signal about the language (it is sub-Turing, or impoverished past the building blocks), not a gap the substrate lives with. This is the admissibility test for adding a language to the catalog, alongside the existing op-layer requirement. The {C, Java, Python} trio of §3.0 is the *minimal* such test: a realization library that discharges across that bracket discharges across the span it brackets, so a new candidate language is admitted by checking it interpolates within the bracket, not by re-running the whole matrix.

## §8: What is already built versus a follow-up

Already built (the substrate this sits on): the op-layer `concept:*` hub and its morphisms (LSP, the cross-language-equivalence explanation, the concept-shapes menagerie); the desugaring layer and `DesugaringEquationMemento` (2026-05-11, PR #601, PR #605); `wp` as formula and the refinement-check discharge (2026-05-13, PR #613); PTP, the transport pipeline and refusal taxonomy (2026-05-12); the Obligation Realizer v2 (2026-05-10); the transport-gap / partial-morphism / lossy-morphism types and the `loss-record` schema (2026-05-14, PR #616, landed); the closure / try-branch / outlives / pin op-layer mementos (2026-05-05/06). This spec's only addition to that substrate's *types* is the `structural_divergence` `loss-record` dimension (§2.4); everything else it adds is new *content* (the abstraction-layer hub nodes, the realization-desugarings) over unchanged machinery.

A follow-up (this spec proposes; it does not implement): everything in §6.1. The natural split is two specs and a multi-PR sub-project under each:

- **This spec** defines the tier model, the `ConceptAbstractionMemento` and `RealizationDesugaringMemento` schemas, the worked `concept:dynamic-dispatch` node, and the mechanics. The abstraction-layer hub catalog (the ~6 nodes with their contract formulas) lands against this spec.

## §9: What this proposal does and does not claim

It claims: that the `concept:*` hub has a tier above the operation layer whose nodes are patterns/protocols/capabilities; that every general-purpose target language realizes every well-formed concept by a possibly-lossy-but-characterized desugaring into that language's operation-layer term; that the abstraction layer is M + N over the same hub topology, with composable per-dimension loss records; that the canonical form of an abstraction is the concept node, every language being a lossy projection; and that a language which genuinely cannot realize a core concept is not a target. It does not claim that the recognition pass is decidable (it is not; it is conservative-or-it-does-not-fire), that the `wp`-formula language is definitely rich enough for every abstraction's contract (open, with a stated fallback), or that the abstraction catalog is complete (it plateaus, like the op catalog; §4 names the starter set, not the whole plateau). It does not build the paradigm layer.

## §10: Why this matters (the closing principle)

The operation layer answered "how do we know two `if`s are the same construct." The abstraction layer answers the harder one: how do we know a C vtable call and a Python method call are the same *thing*, when one is a fixed-offset pointer chain and the other is a name-lookup walk over dicts. The answer is the same answer the rest of the stack gives, one rung up: there is a canonical thing, `concept:dynamic-dispatch`, it is content-addressed, and the two language forms are *both* lossy projections of it, with the loss characterized so precisely that the diff between them is computed rather than tabulated. *Supra omnia, rectum*: a realization that changed the value, or a recognition that was not exact, is not a realization or a recognition, it is a lie, and the substrate refuses to call it one. The premise that makes the whole thing work, that no general-purpose target language cannot express a well-formed concept, is just Turing-equivalence taken seriously at the level a programmer actually thinks at. The languages all sit downstream of the concept. We name the concepts; we do not invent them.
