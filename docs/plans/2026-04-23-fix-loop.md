# `neurallog fix`: Closed-Loop Correctness

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Every bug report becomes a verified fix bundle — code patches, regression tests, complementary defensive artifacts, and (when applicable) a new library principle — atomically applied and proven to close the underlying correctness gap across the entire codebase, not just the reported site.

**Architecture:** Shadow AST (SAST) as the structural substrate. Principle DSL as the query layer over it. LLM at the boundaries (intake parsing, fix proposal, artifact generation). Z3 at every gate (invariant confirmation, overlay verification, coherence checks). Mechanical templates + DSL queries in the hot path. Bundle assembly is all-or-nothing; apply layer is transactional with rollback.

**Tech Stack:** TypeScript, vitest, tree-sitter (existing), ts-morph (existing), Drizzle + better-sqlite3 (existing), Z3 CLI (existing). No new deps.

**Spec:** `docs/specs/2026-04-23-provekit-v2-design.md` — the v2 design doc. This plan implements Phase B (SAST + DSL) and the fix-loop phase as one coherent unit.

---

## Why one phase and not two

Every piece of the fix loop depends on SAST queries or Z3 proofs over SAST-indexed data. Splitting "build SAST" and "build fix loop" as separate phases means the SAST ships with no consumer proving it right, and then the fix loop gets built against a SAST that hasn't been stress-tested by its hardest user. Unified phase lets the SAST's shape evolve in response to the actual queries the fix loop needs. Less guesswork, fewer retrofit rounds.

### The verification contract — not "a few LLM calls"

The goal: **issue in → guaranteed correctness out, verified many ways**. The LLM is fungible and proposes; the mechanical oracles prove. Every bundle passes all of these before apply, and the audit trail records each:

1. **Z3 — bug confirmation.** Negated-goal assertion of the invariant is sat → bug is real.
2. **Z3 — primary fix gate.** Under the SAST overlay with the fix applied, negated goal is unsat → invariant holds.
3. **Z3 — complementary-change gate.** Same check per adjacent site; no adjacent change ships without its own Z3 proof.
4. **Z3 — no-regression check.** Every previously-proven invariant in any affected function is re-proven against the overlay. A fix that breaks a prior proof is rejected.
5. **Z3 — bundle coherence.** The conjunction of every new invariant is checked sat. Bundle that internally contradicts itself is rejected.
6. **Z3 — adversarial validation for new principles.** A different model tier generates false-positive and false-negative examples; Z3 checks the principle's template against each.
7. **Runtime harness — witness replay.** Fix is exercised with the Z3 witness inputs; outcome matches the invariant's claim.
8. **Runtime harness — no-new-gaps check.** Full gap detector runs against the overlay. Any new `gap_reports` row that didn't exist pre-fix kills the bundle.
9. **Mutation-verified regression test.** Test FAILS pre-fix (without the patch applied) and PASSES post-fix. Both directions required.
10. **Full test suite.** Every existing test remains green against the overlay.
11. **SAST structural coherence.** After overlay re-index, every node has valid parent-child edges; the new AST typechecks.
12. **DSL principle evaluator — no silent regressions.** Every principle that matched before either still matches or is accounted for (e.g., the match disappeared because we fixed the bug — expected and recorded). New principles fire only on intended sites.
13. **Gap closure verification — the original signal.** The specific `gap_report` or triggering violation that kicked off the loop is gone post-apply. Proof the loop actually closed what it set out to close.

**Substrate-extension oracles (apply only to substrate bundles — see "Two bundle types" below):**

14. **Migration safety.** SQL migration is non-destructive: `ALTER TABLE ADD COLUMN` and `CREATE TABLE` allowed; any `DROP`, destructive `ALTER`, or column-type narrowing rejected. Schema lints pass (FKs resolve, indexes are valid, no impossible constraints). The rollback path (`DROP` of the new objects + reverse migration) is verified executable against a clone of the live DB.
15. **Cross-codebase regression.** Re-evaluate every existing principle against the substrate-with-new-capability over a representative corpus. The verdict for each principle on each file must match pre/post. A principle whose verdict shifts means the new capability has unintended interaction with the existing library → reject the substrate bundle.
16. **Extractor coverage.** The new extractor's test suite covers positive fixtures (must emit rows), negative fixtures (must not), and edge cases derived from the bug locus + adversarial-validator-generated cousins. All pass under cross-model adversarial validation.
17. **Substrate consistency.** After the extractor runs over the corpus: no orphan FK rows, no NULL-in-NOT-NULL inserts, no PK collisions, declared kind enums hold. The substrate after extraction is queryable and self-consistent.
18. **Principle-needs-capability.** The motivating principle FAILS to compile (or has no possible match) without the new capability, AND succeeds with it. Confirms the capability addition is load-bearing for at least one principle. Gratuitous additions are rejected.

Eighteen independent oracles total: 13 for fix bundles, +5 for substrate bundles. No "LLM said it works, ship it." Every LLM output feeds at least one mechanical verifier; most feed several. The audit trail records every LLM call (prompt + response + model + confidence) and every oracle result (inputs + verdict + output). Bundles are reproducible from the trail alone.

### Two bundle types, one loop

The substrate is not closed. When a bug shape can't be expressed by existing capabilities, the loop extends the substrate using the same machinery — atomic bundle, mechanical oracle gating, library promotion on success. "We can't express this" is a signpost, not a dead end.

- **Fix bundle:** patch + regression test + new principle in existing DSL + library promotion. Oracles 1–13 apply.
- **Substrate bundle:** everything in a fix bundle, PLUS the migration + new capability schema + extractor + extractor tests + capability registry update. Oracles 1–18 apply.

The orchestrator (B5) routes by C6's verdict: if C6 produces a principle in existing capabilities, it's a fix bundle; if C6 returns a principle-plus-capability proposal, it's a substrate bundle. Both kinds apply atomically and both compound the library on success.

The asymmetry that keeps this safe: substrate bundles are rarer (a class of bug exposes a substrate gap once, then never again for that class), more expensive to verify (5 extra oracles, including cross-codebase regression), and pay back across thousands of future analyses. The cost matches the leverage.

### Primitive registries — data at every layer, code only for composition

Hardcoded bug shapes are death. Every layer of the stack has a **registry of primitives** that the LLM can extend through the loop without TypeScript changes. What's in code: the pipeline's composition order. What's in data: everything inside each layer.

**Four layers, four registries:**

1. **SAST capability registry** (`src/sast/capabilityRegistry.ts`, landed in A7a). Each capability is a descriptor: DSL name, Drizzle table, column metadata (sort, nullable, isNodeRef, kindEnum). The compiler resolves capability references via this registry. Adding a capability = schema file + extractor + `registerCapability(...)`. No compiler change.

2. **DSL relation registry** (to land in A7c). Each built-in relation (`before`, `dominates`, `post_dominates`, `data_flow_reaches`, `encloses`, `always_exits`, `branch_reaches`, `mutates`, `literal_value`, `call_arity`, `method_name`, `compound_assignment`) is a descriptor: name, param signature, SQL-template or Drizzle expression-builder that takes bound aliases and emits the relational predicate. The compiler looks up relations through the registry. Adding a relation = `registerRelation({ name, params, compile: (aliases) => sqlFragment })`. No parser change.

3. **Z3 SMT template registry** (already data — `.neurallog/principles/*.json`). Each principle ships an SMT-LIB template keyed by capability bindings. Adding a template = write JSON + its DSL principle. No Z3 layer change.

4. **Unit test pattern registry** (already data — property-test synthesis). Each test pattern is a witness-driven input synthesizer. Adding a pattern = register a synthesizer. No test runner change.

**What stays in code (and why):**
- The pipeline composition: SAST extracts → DSL queries → Z3 verifies → tests confirm. This is the chain of evidence and it never changes.
- The registries themselves (the `Map` that holds the descriptors, the lookup functions, the error-reporting).
- The oracles (1–18) that validate each layer's extensions before they're accepted.

**Why this matters for the guarantee:** when a new bug shape arrives, C6 can provide updates to ANY layer — a new capability, a new relation, a new SMT template, a new test pattern — and those updates flow through the same atomic bundle machinery with the same mechanical verification. Nothing requires a code PR. The LLM's role is always: propose data. The pipeline's role is always: gate the proposal with verified oracles.

This is what "LLM at boundaries, mechanical in the hot path" actually means in practice. The boundaries ARE the registries; the hot path IS the pipeline composition.

### Design decisions locked before task 1

Surfaced here so the plan's tasks don't re-litigate them:

1. **SAST overlay via scratch worktree, not virtual delta.** Fix candidates apply to a temp copy of affected files; SAST re-indexes only those files (incremental via subtree_hash); Z3 runs against the re-indexed state; rollback via discard of the temp copy. Simpler and more honest than maintaining an in-memory delta layer that could diverge from a real apply.
2. **Mutation-verified tests use the same scratch mechanism.** Stash the fix, run the test (must fail), restore the fix, run the test (must pass).
3. **Bundle atomicity via git worktree + stash.** The bundle applies to a scratch worktree, all verification runs there, `git diff` produces the final patch, cherry-picked onto the target branch on success. Rollback is `git worktree remove`.
4. **Adjacent-site discovery is bounded.** SAST query may return 100+ similar sites. Fix loop prioritizes: (1) call sites affected by the fix's signature change, (2) sites inside the same data-flow component, (3) pattern-matched sites ranked by confidence. Cap at N=10 in-bundle; remaining go into `pending_fixes` queue for follow-up bundles.
5. **Confidence tier gates auto-apply.** High-confidence bundles auto-apply; low-confidence bundles produce a PR draft awaiting review. Confidence derived from: LLM self-reported confidence, number of Z3 proofs that held, test count, bundle size (larger = lower confidence), whether the bug class matched an existing validated principle.
6. **Z3 "unknown" is treated as failure.** Any invariant that Z3 can't decide is not a valid proof. The fix candidate is rejected. Unknown is not silently-pass.
7. **LLM calls are logged with prompt + response + model + seed.** Audit trail + reproducibility + debugging. Stored in a new `llm_calls` table linked to bundles.
8. **Data-flow analysis is syntactic for v1.** Same heuristic as Phase B: resolve identifiers to most-recent in-scope assignment. Full type-checker-based alias analysis stays deferred.
9. **Principle confidence starts at "advisory" on creation.** Promoted to "warning" after 3+ successful applications without false-positive reports. Promoted to "blocking" only by explicit curator action. Keeps new principles from polluting CI noise.
10. **The SAST is rebuilt for a file whenever that file's `content_hash` changes.** Incrementality is at the file granularity; subtree_hash inside a file enables fine-grained query invalidation but not rebuild skipping.

**Why:** Nothing else in this plan can run without the SAST being indexable. Start with just nodes + children + hashes so the structural skeleton exists. Capability tables come in A3; edge tables in A4; everything else lands on top.

**Files:**
- Create: `src/sast/schema/nodes.ts`, `src/sast/schema/children.ts`, `src/sast/schema/index.ts`
- Create: `drizzle` migration (next in sequence; check existing count)
- Create: `src/sast/subtreeHash.ts` — deterministic hashing
- Modify: `src/db/schema/index.ts` to re-export SAST schema
- Tests: `src/sast/schema/nodes.test.ts`, `src/sast/subtreeHash.test.ts`

**What the table shapes are:**

Read `docs/specs/2026-04-23-provekit-v2-design.md` — the "Data Model → Structural tables" section spells the exact column-by-column shape. Don't re-derive; implement what's specified.

Key facts:
- `nodes.id` is a TEXT stable across rebuilds of the same subtree (hash-based or UUID-based; hash is simpler because subtree-hash invalidation naturally ties to node identity churn).
- `nodes.subtree_hash` is `sha256(node.getText())` — no canonicalization per prior design decision.
- `nodes.file_id` FKs to a `files` table. Create `files` if not already present.
- `node_children` has composite PK `(parent_id, child_id)` + `child_order` column.

**What to verify:**

Insert and query: a single node, a file with multiple nodes and parent-child edges, a subtree hash matching `sha256(text)`. No actual ts-morph integration yet — this is raw schema testing.

**Commit:** `sast: core tables (nodes, children, subtree hash) via Drizzle`

### Task A2: SAST builder — ts-morph to nodes + children

**Why:** Now that tables exist, populate them from real source. This task does NOT populate any capability tables — just nodes and children and subtree hashes. Capabilities come in A3.

**Files:**
- Create: `src/sast/builder.ts` — export `buildSASTForFile(db, filePath): SASTBuildResult`
- Create: `src/sast/builder.test.ts`

**The API:**

```ts
export interface SASTBuildResult {
  fileId: number;
  rootNodeId: string;
  nodeCount: number;
  rebuilt: boolean;  // false if content_hash matched existing — skipped
}

export function buildSASTForFile(db: Db, filePath: string): SASTBuildResult;
```

**Mechanism:**

1. Read file, compute `content_hash = sha256(bytes)`.
2. If `files.content_hash` for this path already matches, return `rebuilt: false` and skip. Skip this optimization for now if it complicates atomicity; revisit in A6.
3. Parse via ts-morph. Walk the AST. Per visit: assign a stable `node.id` (proposal: `sha256(file_id + start_position + end_position + subtree_hash).slice(0, 16)`), compute subtree_hash, insert into `nodes`, insert edge into `node_children`.
4. Transactional: wrap in `db.transaction(async tx => ...)` so partial failure doesn't leave orphan rows.

**What to verify:**

A fixture function source → node count matches hand-counted ts-morph walk. Root node `subtree_hash` matches `sha256(source)`. Children relationships respect `child_order` (in-order tree walk reproduces original source).

**Hazards:**
- ts-morph's parent-child relationship doesn't always correspond to how we want to structure capability queries later. Capture the raw tree-sitter-style parent-child; capabilities build on top.
- Unicode source files — `.getText()` should handle, but verify with one fixture.
- Don't walk into node_modules imports. The parse happens per file; cross-file edges come later.

**Commit:** `sast: builder populates nodes + children from ts-morph AST`

### Task A4: Data-flow edges (syntactic def-use)

**Why:** Principles need data-flow. Gap detection needs data-flow. Fix-loop's complementary-site discovery needs data-flow. Every consumer downstream needs this.

**Files:**
- Create: `src/sast/dataFlow.ts` — the extractor
- Create: `src/sast/schema/dataFlow.ts` — schema (already in v2 spec)
- Tests: `src/sast/dataFlow.test.ts`

**The algorithm (syntactic heuristic per v2 spec decision):**

Per function:
1. Identify all identifier references in the function body (use ts-morph or tree-sitter).
2. For each reference, resolve to its most-recent in-scope declaration/assignment preceding it (lexical scope walk).
3. Emit an edge `from_node = <declaration/assignment>`, `to_node = <use>`, `slot = "read"`.
4. For binary expressions specifically, emit slot-labeled edges: `slot = "lhs" | "rhs" | "denominator" | "arg[0]"` so principles can query by slot position.

**What to verify:**

Fixture `function f(a, b) { const q = a / b; return q; }` produces:
- edge: `a` (decl) → `a / b` (use, slot=lhs)
- edge: `b` (decl) → `a / b` (use, slot=rhs or slot=denominator)
- edge: `q` (decl) → `return q` (use, slot=read)

**Don't do:**
- No alias analysis. If `const x = a; const y = x; y / b;` — the edges are `a→x→y→use`, step by step. Not `a → use` directly. That's an intentional limitation per v2 design.
- No cross-file edges. Imports are out of scope for v1 data-flow.

**Commit:** `sast: syntactic def-use data-flow edges`

### Task A5: Dominance + post-dominance

**Why:** Principles that express "no guard dominates this division" or "every exit passes through this cleanup" need dominance. Fix-loop uses dominance to identify regions where invariants hold post-fix. Both directions needed.

**Files:**
- Create: `src/sast/dominance.ts`
- Create: schemas `dominance.ts`, `postDominance.ts`
- Tests: `src/sast/dominance.test.ts`

**The algorithm:**

Per function's CFG (reconstructed from `decides`, `iterates`, `throws` capabilities):
1. Build CFG: nodes are SAST nodes, edges are "executes before" relationships.
2. Compute dominance via standard iterative dataflow (Cooper/Harvey/Kennedy fast algorithm is overkill for MVP; the obvious O(N²) fixpoint is fine).
3. Compute post-dominance same way but on the reverse CFG.
4. Materialize transitive closures to `dominance` and `post_dominance` tables.

**What to verify:**

Fixture with explicit branches:
```ts
function f(x) {
  if (x > 0) return x;  // A: guard check
  throw new Error();     // B: always throws
  return 0;              // unreachable
}
```

- `dominance`: function-entry dominates A, A dominates the return, B is dominated by A's else-branch.
- `post_dominance`: the return post-dominates A's then-branch; the throw post-dominates A's else-branch.

**Don't do:**
- Don't try to reason about dynamic behavior (async, exception flow, etc.). CFG is static and conservative.
- Don't include exception edges (catch clauses) in the CFG for v1 — they're explicit via `throws` capability and future work.

**Commit:** `sast: dominance + post-dominance via CFG + transitive closure`

### Task A6: Subtree-hash-based incremental re-index

**Why:** The fix loop will re-index files repeatedly (once per fix-candidate verification, once per final apply, once per regression test mutation). Full re-parse + re-walk every time is untenable. Skipping re-index when `content_hash` unchanged is the easy optimization; at the SAST-node level, subtree-hash enables fine-grained invalidation of only affected subtrees in very large files.

**Files:**
- Modify: `src/sast/builder.ts` — check `content_hash` before re-parsing; support `reindexFile(db, filePath)` entry point
- Create: `src/sast/incremental.ts` — subtree-level invalidation (optional for MVP; skip if time-pressed, full file re-index is acceptable)
- Tests: `src/sast/incremental.test.ts`

**Mechanism:**

1. `buildSASTForFile` computes content_hash first. If matches DB, skip.
2. `reindexFile(db, filePath)` forces re-parse even if hash matches (for use after hot edits that might have changed the file between analyses).
3. (Optional) If content_hash differs but most subtree_hashes match, only re-walk subtrees whose hash changed. Update `nodes` rows, preserve `nodes.id` for unchanged subtrees. This is a meaningful optimization only on huge files; skip for MVP if complex.

**What to verify:**

- Second call to `buildSASTForFile` with unchanged source returns `rebuilt: false` and doesn't insert duplicate rows.
- Editing a file (new content_hash) and calling `reindexFile` produces updated rows.

**Commit:** `sast: incremental re-index via content_hash and subtree_hash`

### Task A7: Principle DSL parser + Drizzle compiler

**Why:** Principles need a query language. The spec defines the shape: Datalog-flavored, compiles to Drizzle query-builder calls. This is the boundary between human-authored (or LLM-generated) principle definitions and the mechanical evaluator that runs them.

**Files:**
- Create: `src/dsl/parser.ts` — hand-rolled recursive-descent or minimal PEG
- Create: `src/dsl/ast.ts` — DSL AST types (match clauses, where predicates, require clauses, report block)
- Create: `src/dsl/compiler.ts` — AST → Drizzle query builder
- Create: `src/dsl/relations.ts` — built-in relations: `before`, `dominates`, `post_dominates`, `data_source`, `data_flow_reaches`, `encloses`, `always_exits`, `branch_reaches`, `mutates`, `literal_value`, `call_arity`, `method_name`, `compound_assignment`
- Create: `src/dsl/evaluator.ts` — execute a compiled principle against the SAST; write matches to `principle_matches` + `principle_match_captures`
- Tests: `src/dsl/parser.test.ts`, `src/dsl/compiler.test.ts`, `src/dsl/evaluator.test.ts`

**The DSL surface (per v2 spec):**

```
principle division-by-zero {
  match
    $div: node where arithmetic.op == "/"
    $den: node where arithmetic.rhs_of($div) == $den
  require no $guard: zero_guard($den) before $div
  report violation {
    at $div
    captures { division: $div, denominator: $den }
    message "division denominator may be zero"
  }
}

predicate zero_guard($var: node) {
  match $guard: node where narrows.narrowing_kind == "literal_eq"
                        and narrows.target == $var
}
```

**Compiler responsibilities:**

1. Resolve capability references (`arithmetic.op`) via a **runtime capability registry** — `Map<capabilityName, { table, columns: { name → { sort, nullable, kindEnum? } } }>`. The registry is populated at startup by each capability schema file calling `registerCapability(name, descriptor)`. **The compiler MUST NOT hardcode the capability list.** Adding a new capability is pure data: schema file + extractor + registration call. This is non-negotiable: substrate self-extension (oracles 14–18) depends on capabilities being a runtime-extensible set.
2. Resolve built-in relations (`before`, `dominates`) to join patterns against `dominance` / `data_flow_transitive`. Built-in relations ARE part of the language and ship hardcoded; capability tables are NOT.
3. Compose match clauses into a single Drizzle query with joins.
4. Translate `require no` to `notExists(subquery)`.
5. Emit an `INSERT INTO principle_matches` at the end with captures.

**Compile-time checks:**
- Capability column typos (reject at compile time with suggestion)
- Enum value typos
- Unknown built-in relations
- Stratification: predicate call graph must be acyclic through negations
- Binder types: a `captures { x: $foo }` where `$foo` doesn't resolve to a node-typed binding fails

**What to verify:**

- Parser round-trips the division-by-zero principle: source → AST → (compiler) → Drizzle query → (executor) → matches.
- Compile-time error cases: typo in `arithmetic.opp` produces "unknown column; did you mean `op`?"
- Running the compiled query against a populated SAST produces the expected matches.

**Commit:** `dsl: parser + compiler + evaluator + built-in relations`

```ts
  paramTypes: ("node" | "literal" | "sort")[];     // for type-checking varRef types at compile time

   * Given the bound Drizzle aliases for each param (resolved by the compiler
   * from the DSL varRefs), emit a SQL fragment that asserts the relation.
   * The fragment is combined into the WHERE clause of the main match query.
   */

}

export function registerRelation(d: RelationDescriptor): void;
export function getRelation(name: string): RelationDescriptor | undefined;

```

### Task C1: Invariant formulator

**Why:** The fix loop's first real decision: what IS the bug, in formal terms? The LLM reads the bug and its SAST context, then writes an SMT-LIB claim that should hold post-fix. Z3 then proves the CURRENT code VIOLATES the claim (confirming the bug is real), and later Z3 will prove the POST-FIX code satisfies it (confirming the fix is correct).

**Files:**
- Create: `src/fix/invariant.ts`
- Tests: `src/fix/invariant.test.ts`

**The API:**

```ts

  smtLib: string;
  bindings: SmtBinding[];  // constant → SAST node
  rationale: string;
  llmConfidence: number;
}

  signal: BugSignal,
  locus: BugLocus,
  db: Db,
  llm: LLMProvider,
): Promise<InvariantClaim>;

  claim: InvariantClaim,
  db: Db,
): Promise<{ violated: boolean; z3Verdict: "sat" | "unsat" | "unknown"; z3Output: string }>;
```

### Task C6: Principle candidate generator (with substrate-extension path)

**Why:** If the bug class is codifiable and new to the library, this is the LLM's one shot to encode it. C6 produces ONE of three outcomes: a principle in existing capabilities, a principle PLUS a new capability proposal, or `null` (bug not codifiable). The substrate-extension path is non-negotiable — without it, we cap the system at the capabilities the original designer imagined, which is the bug we are explicitly building against.

**Files:**
- Create: `src/fix/principleGen.ts`
- Create: `src/fix/capabilityGen.ts` — proposes new capability schema + extractor when no existing capability suffices
- Tests: `src/fix/principleGen.test.ts`, `src/fix/capabilityGen.test.ts`

**The API:**

```ts
export type PrincipleCandidate =
  | {
      kind: "principle";
      name: string;
      dslSource: string;
      smtTemplate: string;
      teachingExample: { domain: string; explanation: string; smt2: string };
      adversarialValidation: { validatorModel: string; result: "pass" | "fail"; evidence: string }[];
      latentSiteMatches: { nodeId: string; file: string; line: number }[];
    }
  | {
      kind: "principle_with_capability";
      name: string;
      dslSource: string;
      smtTemplate: string;
      teachingExample: { domain: string; explanation: string; smt2: string };
      adversarialValidation: { validatorModel: string; result: "pass" | "fail"; evidence: string }[];
      latentSiteMatches: { nodeId: string; file: string; line: number }[];
      capabilitySpec: CapabilitySpec;  // see below
    }
  | null;

export interface CapabilitySpec {
  capabilityName: string;             // "node_xyz"
  schemaTs: string;                    // contents of new src/sast/schema/capabilities/xyz.ts
  migrationSql: string;                // contents of drizzle/00NN_xyz.sql
  extractorTs: string;                 // contents of new src/sast/capabilities/xyz.ts (or addition to extractor.ts)
  extractorTestsTs: string;            // contents of test file
  registryRegistration: string;        // the line that registers the capability in the runtime registry
  positiveFixtures: { source: string; expectedRowCount: number }[];
  negativeFixtures: { source: string; expectedRowCount: 0 }[];
  rationale: string;                   // why existing capabilities can't express the principle
}

export async function generatePrincipleCandidate(
  signal: BugSignal,
  invariant: InvariantClaim,
  fixCandidate: FixCandidate,
  db: Db,
  llm: LLMProvider,
): Promise<PrincipleCandidate>;
```

**Mechanism:**

1. Classify: is this bug class already covered by an existing principle? (Run existing principles against the invariant's fixture; if one matches, return `null`.)
2. LLM (primary): try to propose the principle in the EXISTING DSL using EXISTING capabilities. If the LLM can express it, attempt compile.
3. If compile fails because a referenced capability/column doesn't exist, OR if the LLM reports "I cannot express this with the available capabilities," route to the capability-generation path:
   a. LLM derives the missing predicate from the bug locus (`capabilityGen.ts`).
   b. LLM proposes `CapabilitySpec`: schema columns, extractor logic, fixtures, rationale.
   c. The capability spec is graded by the substrate-extension oracles BEFORE the principle is finalized:
      - Oracle 14 (migration safety): lint the SQL.
      - Oracle 16 (extractor coverage): compile and run the extractor against fixtures; verify positive emits rows, negative does not.
      - Oracle 17 (substrate consistency): run extractor against a corpus; verify no FK orphans, no PK collisions.
      - Oracle 18 (principle-needs-capability): show the principle FAILS to compile without the new capability and SUCCEEDS with it.
   d. If any of these fail, the substrate-extension path aborts and the LLM is asked to retry with the failure feedback.
4. Compile the principle (now with the new capability registered if applicable).
5. Adversarial validation — use a DIFFERENT model tier. Generate 5 false-positive code examples that should NOT trigger the principle; 5 false-negative bugs that SHOULD trigger. Run the compiled query; principle must reject false positives AND catch false negatives.
6. If adversarial passes, run the compiled principle against the FULL SAST (with the new capability extracted across the corpus if applicable). Collect latent-site matches.
7. Return the candidate. The orchestrator decides whether to include it in the bundle.

**Cut list:**
- Don't add to the principle library or register the capability yet. That's D3. This task produces the candidate package only.
- Don't run adversarial forever — 5 examples per direction is the v1 gate.
- Don't try to derive capabilities from thin air — the bug locus constrains what predicate is missing. The LLM does pattern-completion against the bug, not free-form table design.

**Commit:** `fix: principle candidate generator with cross-model adversarial validation and substrate-extension path`

## Section D: Bundle + Apply + Learn

### Task D1: Bundle assembly + coherence verification (two bundle types)

**Why:** All the artifacts from C1-C6 are independently verified. The bundle combines them and checks they cohere. Bundles come in two types — fix and substrate — distinguished by whether C6 returned a `principle_with_capability`. Both apply atomically; substrate bundles run additional oracles before assembly succeeds.

**Files:**
- Create: `src/fix/bundle.ts`
- Create: `src/db/schema/fixBundles.ts` — `fix_bundles`, `fix_bundle_artifacts`, `llm_calls`, `pending_fixes` tables
- Tests: `src/fix/bundle.test.ts`

**The API:**

```ts
export type BundleType = "fix" | "substrate";

export interface FixBundle {
  bundleId: number;
  bundleType: BundleType;
  bugSignal: BugSignal;
  plan: RemediationPlan;
  artifacts: {
    primaryFix: FixCandidate;
    complementary: ComplementaryChange[];
    test: TestArtifact | null;
    principle: PrincipleCandidate | null;
    startupAsserts: CodePatch[];
    observabilityHooks: CodePatch[];
    documentation: { path: string; content: string }[];
    // Substrate-bundle artifacts (null for fix bundles)
    capabilitySpec: CapabilitySpec | null;  // present only when bundleType === "substrate"
  };
  coherence: {
    sastStructural: boolean;
    z3SemanticConsistency: boolean;
    fullSuiteGreen: boolean;
    noNewGapsIntroduced: boolean;
    // Substrate-only oracle results (null for fix bundles)
    migrationSafe: boolean | null;
    crossCodebaseRegression: boolean | null;
    extractorCoverage: boolean | null;
    substrateConsistency: boolean | null;
    principleNeedsCapability: boolean | null;
  };
  confidence: number;
  auditTrail: AuditEntry[];
}

export async function assembleBundle(
  parts: BundleParts,
  db: Db,
): Promise<FixBundle>;
```

**Coherence checks (all bundles — oracles 1–13):**

1. **Structural**: create a single overlay with ALL patches applied. Re-index. Check parent-child edges intact.
2. **Z3 semantic**: conjoin every new invariant. Check-sat. Must be sat. If unsat, reject.
3. **Full suite**: run the entire test suite against the overlay. All must pass. Run the principle evaluator — every existing principle that matched before the bundle must still match (or the bundle legitimately fixes it).
4. **No new gaps**: run the gap detector. Any new `gap_report` row → reject.

**Substrate bundle additional checks (oracles 14–18) — run BEFORE the standard 1–13:**

5. **Migration safety (oracle 14)**: lint the migration SQL — only `ALTER ADD COLUMN` and `CREATE TABLE` allowed; no destructive ops. Verify rollback path is executable on a clone.
6. **Apply migration to overlay DB**, then run the new extractor over the corpus.
7. **Cross-codebase regression (oracle 15)**: re-evaluate every existing principle on the post-migration substrate. Any verdict shift → reject.
8. **Extractor coverage (oracle 16)**: positive fixtures emit rows, negative fixtures don't. Cross-model adversarial validation passes.
9. **Substrate consistency (oracle 17)**: no FK orphans, no PK collisions, kind enums hold across the corpus.
10. **Principle-needs-capability (oracle 18)**: confirm the new principle FAILS to compile against the pre-migration substrate AND succeeds post-migration.

If ANY substrate oracle fails, reject the bundle BEFORE running oracles 1–13. (No point checking the principle's coherence if the substrate it depends on isn't sound.)

Aggregate confidence from component confidences, with dampening for bundle size and additional dampening for substrate bundles (broader blast radius).

**Cut list:**
- Don't merge bundles. One signal → one bundle.
- Don't try to reconcile contradictions between complementary changes — reject and ask orchestrator to iterate.
- Don't allow a substrate bundle to skip ANY of oracles 14–18 because the cost of a bad capability in production is N orders of magnitude higher than a bad principle.

**Commit:** `fix: bundle assembly with two bundle types and substrate-extension oracles`

### Task D2: Apply layer — transactional

**Why:** The bundle's artifacts land on the real branch, atomically. Either all of it applies and the gap closes, or none of it applies. Git worktree + cherry-pick provides the atomicity.

**Files:**
- Create: `src/fix/apply.ts`
- Tests: `src/fix/apply.test.ts`

**The API:**

```ts
export interface ApplyResult {
  applied: boolean;
  commitSha?: string;
  closedGaps: string[];  // gap_report IDs that are gone post-apply
  failureReason?: string;
}

export async function applyBundle(
  bundle: FixBundle,
  targetBranch: string,
  db: Db,
  options: { autoApply: boolean; prDraftMode: boolean },
): Promise<ApplyResult>;
```

**Mechanism:**

1. Create a fresh worktree off `targetBranch`.
2. **If substrate bundle:** apply the migration FIRST to the live DB inside a transaction. (Schema must exist before extractor can populate. Rollback path armed: a `DROP` script ready to run if any later step fails.)
3. **If substrate bundle:** copy the new capability schema file, extractor file, registry registration into the worktree.
4. Apply every code patch in the bundle to the worktree (primary fix + complementary + asserts + observability).
5. Re-index the SAST for all affected files (substrate bundle: reindex EVERY file in the corpus, since the new capability needs to be extracted everywhere).
6. Re-run the analyze pipeline. Diff the gap_reports.
7. Assert the triggering gap is closed. If not, fail, ROLLBACK the migration if substrate bundle, report.
8. Commit in the worktree with a structured message referencing the bundle ID. Substrate bundles include the schema file and extractor in the commit.
9. If `autoApply`: cherry-pick onto `targetBranch`. Update the live DB. Register the new capability in the runtime registry on next process start (or hot-register if supported).
10. If `prDraftMode`: keep the worktree; produce a patch file + PR draft body + (if substrate) the migration script. Hand all to the user.
11. Cleanup worktree on success.

**Substrate-bundle rollback semantics:**
- If apply fails AFTER migration but BEFORE the bundle commits, run the rollback script (DROP the new objects). The live DB returns to pre-migration state.
- The new capability extractor never ran against any pre-existing committed code, so no orphan rows.
- Audit trail records the rollback as a first-class event.

**Cut list:**
- Don't push to any remote.
- Don't force-push. Don't rewrite history. Only cherry-pick onto the current branch head.
- On failure mid-apply, the worktree stays (for debugging); the target branch is untouched; substrate-bundle migrations are rolled back.

**Commit:** `fix: transactional apply via scratch worktree, substrate-aware, with rollback + gap closure verification`

### Task D3: Learn — library updates post-apply

**Why:** The bundle applied; now the system gets smarter. New principles enter the library, gap-agent refinements apply, confidence updates persist, latent sites queue for follow-up.

**Files:**
- Create: `src/fix/learn.ts`
- Modify: `src/db/schema/principles.ts` (already exists in some form) — add confidence_tier column, validation_history
- Tests: `src/fix/learn.test.ts`

**The API:**

```ts
export interface LearnResult {
  principleAdded: string | null;
  confidenceUpdates: { principleName: string; newTier: string; reason: string }[];
  gapAgentRefinements: string[];
  pendingFixesEnqueued: number;
}

export async function learnFromBundle(
  bundle: FixBundle,
  applyResult: ApplyResult,
  db: Db,

```

**Mechanism:**

1. If bundle included a validated `PrincipleCandidate`:
   - Insert into `principles` table with confidence_tier = "advisory".
   - Write the DSL source to `.neurallog/principles/<name>.dsl`.
   - Enqueue a re-run of the principle against all files.
2. **If substrate bundle:** the new capability is now part of the substrate library.
   - Write the capability schema file to its permanent location.
   - Write the extractor + tests to permanent locations.
   - Append the registry registration to the capability index.
   - Record the capability addition as a first-class event in the audit trail (which bug motivated it, which oracle results it survived).
   - Future analyses on this codebase (and any codebase that takes this provekit version) get the capability automatically. The substrate compounds at the platform level.
3. For every principle that existed before but matches the now-applied fix (meaning it SHOULD have caught this bug and didn't), annotate its validation history with a false-negative mark.
4. For every new latent site identified during principle generation, insert into `pending_fixes` table with the bundle as origin.
5. Record the full audit trail (every LLM call, every Z3 check, every SAST query, every capability registration) linked to the bundle for reproducibility.

**Cut list:**
- Don't auto-broadcast to other codebases. Cross-codebase sync is a separate phase.
- Don't auto-promote principle tiers. Confidence promotion is manual (via a curator command) for v1.
- Don't auto-revoke a capability. A regretted capability requires a manual curator action; the cost-asymmetry is too high for automated removal.

**Commit:** `fix: learn layer — principle persistence, confidence updates, pending-fixes queue`
