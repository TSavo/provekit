# ProvekIt papers

This directory contains paper-grade arguments. These are not docs (how to use the protocol) and not specs (the formal definition of the protocol). They are sustained arguments about what the protocol *is*, why its design choices are load-bearing, and what its consequences are.

Each paper is intended to be cite-able in academic and industry contexts.

## Index

The papers are an inductive ladder. Paper 1 is the pitch, paper 2 is the formal spec, and papers 3 through 23 are the After-X rungs: given what the prior paper established, here is what changes. Read them in order, because later rungs assume the framing earlier ones built.

1. **[The Whitepaper](01-whitepaper.md)**: Executive summary. What ProvekIt is, why it matters, who it is for, the trojan horse, the cypherpunk lineage. The trust-depth knob; the three-CLI surface; the install path. The shortest path from "I have heard of this" to "I understand the move."

2. **[The Bluepaper](02-bluepaper.md)**: Formal protocol specification. Theorem statements with proofs; the canonical IR grammar in EBNF; the constant-size verification theorem with hypotheses H1-H5; the `memcmp` line shown verbatim; the cryptographic-minimum claim; every spec referenced by content hash. Closes with a runnable verification: compute the catalog CID locally, and the bluepaper has just verified its own authority. The paper is a formal baseline; the current v1.6.3 catalog and extension CIDs live in [../reference/cids.md](../reference/cids.md) and [../reference/protocol-extensions.md](../reference/protocol-extensions.md).

3. **[After Bluepapers: Substrate, Not Blockchain](03-substrate-not-blockchain.md)**: The proofchain/blockchain adjacency argument. Contains the formal lemma that proof validity does not require distributed consensus, while distributed timestamp-server claims do. Also covers §10 closure-by-composition, §11 the address as multi-dimensional, and §12 the pin as a tuple. The four pinning dimensions (`contractCid` / `contractSetCid` / `attestationCid` / bundle file CID), each a different content projection. The rank-N tuple discipline: a pin must match the rank of the assertion it attests; collapsing a rank-2 relation into rank-1 leaks the discarded axis as drift. This is the architectural foundation the v1.4 specs operationalize.

4. **[After the Standardization Question: The Vertical Stack and the Road to Standardization](04-vertical-stack-and-standardization.md)**: A `.proof` and a chain of formally verified software from quantum physics to bytecode are 1:1 identical at the data-structure level. The standardization roadmap through DO-178C, Common Criteria EAL5+, ISO 26262, FDA SaMD, FedRAMP, IEC 62304, NIST SSDF, SLSA, EU Cyber Resilience Act. 5/10/15/20-year horizons. Runnable counterpart: [`menagerie/bridgeworks`](../../menagerie/bridgeworks/README.md).

5. **[After Single-Party Transport: Witness Pluralism and Jurisdiction-Neutral Transport](05-witness-pluralism-and-jurisdiction-neutral-transport.md)**: Why the substrate moves through any pipe (HTTP, IPFS, dead-drops) and why federation across jurisdictions doesn't require any single party. The transport layer's design as an explicit non-feature, and what that buys.

6. **[After Reputation: Software as Federated Truth-Claims](06-after-reputation-software-as-federated-truth-claims.md)**: The consequence of shipping the protocol. Why the substrate replaces reputation as the load-bearing trust mechanism in supply chains, engineering practice, liability, and the closed/open dichotomy. The substrate is the diplomatic protocol between every truth-claim about software ever made. Runnable counterpart: [`menagerie/supply-chain-rails`](../../menagerie/supply-chain-rails/README.md).

7. **[After Verification: Bug Classes as Missing Edges in the Federated Proof Substrate](07-after-verification-bug-classes-as-missing-edges.md)**: The deeper consequence. Once droppers close the loop with lifters over weakest-precondition propagation, leaf-discharge bug classes become structurally impossible. Contains a constructive theorem (Structural Elimination of Leaf-Discharge Bug Classes) with proof by induction on data-flow path length. Articulates the substrate's algebraic shape (thin Heyting category over content-addressed predicates), the completeness lemma (`Allocations × Reads` is enumerable and exhaustive), and the generative-completion property (the substrate computes what is missing and writes the code that supplies it). Cousot 1977 plus content-addressing plus federation, lifted to a planet-scale proof DAG. Runnable counterpart: [`menagerie/bug-zoo`](../../menagerie/bug-zoo/README.md).

8. **[After Types: How I Learned to Stop Logging and Trust the Invariant Solver](08-after-types-stop-logging-trust-the-invariant-solver.md)**: The developer-workflow consequence. Types and forensic logs survive, but lose their load-bearing correctness role once invariant proof is a federated substrate. Types become editorial scaffolding; logs remain operational observability. The wall against leaf-discharge bugs moves to content-addressed invariant edges.

9. **[After Universal-Language Ambitions: Lossy Boundary Compression (ProofIR Is Universal Because It Forgets)](09-lossy-boundary-compression.md)**: The universality argument for ProofIR. ProofIR is not a universal language for re-expressing every implementation detail; it is universal over contract boundaries. Lifters may discard implementation texture while preserving obligations, making cross-language, cross-framework, cross-time equivalence possible and turning LLM output into an admissibility-search problem. Runnable counterpart: [`menagerie/bug-zoo`](../../menagerie/bug-zoo/README.md) (Bug Zoo grounds both papers 07 and 09; §10 of paper 09 names it as receipt infrastructure).

10. **[After Protocol Specs: How Protocols Actually Evolve](10-after-protocol-specs-how-protocols-actually-evolve.md)**: Protocol evolution as data. Protocols have contract boundaries too: grammars, state machines, invariants, profiles, conformance witnesses, refusals, and migration edges. A protocol version becomes a signed content-addressed graph; a protocol change becomes a witnessed edge from old root to new root. Runnable counterpart (planned): [`menagerie/protocol-switchyard`](../../menagerie/protocol-switchyard/README.md).

11. **[After Commits: Proof-Carrying Change as p -> q](11-after-commits-proof-carrying-change.md)**: The commit as semantic transition. A commit becomes `tree + parent + message + .proof root`, where `.proof` is the typed claim root: preservation receipts for same behavior, fix receipts for changed behavior, refusal receipts for explicit unknowns, and edge compression carrying the proof tree as `p -> q`. CI solves the bootstrap by signing `CommitProofBinding(commitCid, proofRootCid)` before native commit-object support exists. Runnable counterpart (planned): [`menagerie/change-station`](../../menagerie/change-station/README.md).

12. **[After Languages: How ProofIR Represents Every Language](12-after-languages-how-proofir-represents-every-language.md)**: The substrate's first axiom *Supra omnia, rectum* binds the substrate's outputs but until now did not bind the substrate's own production mechanism: five language ports of the same algorithm with no canonical reference and no mechanically-detectable drift. This paper closes the inconsistency: algorithms become content-addressed mementos, language ports become verifiable refinement-claims, the substrate hosts its own production. Contains nine lemmas (federation by CID, soundness preservation, drift detection, syntactic algorithm identity, compositional lifters, language closure, Cousot equivalence, algebraic-effects embedding, substrate self-hosting). Companion spec: [Algorithm Memento Protocol (AMP) v0.1.0](../../protocol/specs/2026-05-09-algorithm-memento-protocol.md).

13. **[After Grammars: Programming Languages as Content-Addressed Algebras](13-after-grammars-programming-languages-as-content-addressed-algebras.md)**: Universal algebra and category theory established that programming language grammars ARE algebras. This paper makes those algebras content-addressed and federated in the substrate's catalog. Every language is a finite signature (sorts + operations + equations + effect signatures); compilation is a homomorphism morphism memento; cross-language verification is composition of morphisms. Contains nine lemmas (signature CID identity, morphism composition, initial-algebra universality, cross-language soundness via homomorphism, compilation correctness as homomorphism discharge, effect signatures as embedded Lawvere theories, equational reasoning via catalog equations, substrate completeness for algebraic languages, language design as algebra construction). Companion spec: [Language Signature Protocol (LSP) v0.1.0](../../protocol/specs/2026-05-09-language-signature-protocol.md).
14. **[After Trust: The Universal Correctness Bundle](14-after-trust-the-universal-correctness-bundle.md)**: Closes the 04/06/07/09/12/13 arc with the `.proof`-as-universal-guarantee thesis: the substrate's deliverable is a constant-size, locally-verifiable correctness bundle whose verifier reduces to CID recomputation, byte comparison, and signature checking. It states the bedrock thesis that the deployed runtime layer is overwhelmingly C and C++ source already reachable by the existing lifter chain, the supply-graph thesis that package registries lift as content-addressed dependency graphs cross-linked to native source, the Bridge collapse that most cross-language runtime behavior reduces to finite calling-convention shims, and the five plateau axes by which the catalog exhausts for a snapshot. The six lemmas cover bedrock dominance, Bridge minimality, plateau finiteness, domain agnosticism, constant-size verification under composition, and CVE blast-radius as SELECT. Runnable counterpart (planned): the source-ingestion pipeline producing populated catalogs.

15. **[After Civilization: Why the Author Doesn't Matter](15-after-civilization-why-the-author-doesnt-matter.md)**: The capstone of the ladder. Paper 14 moved trust off the consumer; this paper moves identity off the producer. Once verification is local and content-addressed, the author of a `.proof` is epistemically irrelevant: the verifier's accept/reject path references a key, receipts, and a local policy, never a person. This generalizes Satoshi's move from money to correctness, and goes further: where Satoshi chose to disappear, the substrate makes the author's absence structurally costless. The author can be anonymous, dead, discredited, coerced, or erased, and existing claims remain exactly as verifiable. Contains four lemmas (author-independence of verification, costless disappearance, accountability relocation not dissolution, and self-demonstration: the paper's own thesis is verified by recomputing its CID without trusting its byline). It is, by design, the "Satoshi" paper of the series.

16. **[The Universal Address Space: After Portability](16-after-portability-the-universal-address-space.md)**: The capstone after the capstone. Papers 12 and 13 made algorithms and languages content-addressed algebras; paper 14 named the `.proof` deliverable; paper 15 made the author irrelevant. This paper names the universal object directly: the algebra is the portable thing, and content-addressing makes each algebra element an artifact. It reframes compiler IR history by showing why JVM bytecode, .NET CIL, LLVM IR, GIMLE, WASM, and C as portable assembly were useful but not universal. It frames the address space as colimits of lift diagrams, monoidal under conjoining, and as a self-organizing map without loss where discharged morphisms collapse true sameness to one CID. It also reconciles paper 9 by distinguishing the faithful term stratum from the lossy contract stratum, cites ORP v0.2's `compile` mode, round-trip theorem, and N plus M factoring, and grounds the claim in the C11-signature and foo-algebraic-shape menagerie exhibits. The anchor is: "JNZ isn't portable. The algebra is."

17. **[After Babel: We Speak in Vectors Now](17-after-babel-we-speak-in-vectors-now.md)**: The capstone of the language thread. Paper 12 made ProofIR represent every language; paper 13 made the languages content-addressed algebras; paper 16 named the address space. This paper draws the conclusion: it is a conclusion about naming, not software. A symbolic language has CIDs for symbols; the symbol *is* its canonical bytes. What Leibniz's *characteristica universalis* was missing was never the calculus of reasoning but a canonicalization function (JCS plus a cryptographic hash) that closes the metalanguage gap by deleting the metalanguage. So this is discovery, not invention: the address of an operation was always a mathematical fact, merely not yet evaluated. A vector-name is the only kind of name with coordination-free reference (two parties who never met arrive at the same address for the same structure), which is why natural language was never a candidate for the language of mathematics and a CID is. Programming languages, proof assistants, instruction sets are dialects: necessary, lossy, emphasized surfaces, made commensurable for the first time by a common address space, with the renaming morphisms as the discharged proofs the maps commute. The slogan: "We name the thing by vector. Everyone else is speaking a dialect." It carries the same self-demonstrating epilogue lineage as paper 15: the byline is a dialect; the signature is the courtesy; the CID is the name.

18. **[After Static Analysis: How Homomorphic Algebra Saved Humanity (and Software Development)](18-after-static-analysis-how-homomorphic-algebra-saved-humanity.md)**: The consumer-side gradient. Paper 07 made bug-class elimination a theorem; paper 13 made languages content-addressed algebras; paper 16 named the operation-tier address space; paper 17 named what we do when we discharge morphisms over it. This paper takes the next rung: the recurring abstractions of imperative programming (`concept:option`, `concept:dynamic-dispatch`, `concept:iterator`, `concept:closure`, `concept:exception`, `concept:reference`, `concept:generic-instantiation`) become first-class content-addressed mementos in a federated catalog. Static analysis stalled for fifty years not for lack of cleverness but for lack of a universal address space for abstractions. Each `(concept, language)` realization is a content-addressed cell with a multidimensional loss-record; cross-language transport is M+N composition through the concept hub, not M×N. The three-key composition `k(k'(k''(I))) = t` is the 3DES extension of paper 16's lift/realize cipher pair: lift, operation-tier morphism, abstraction-tier realization, each a content-addressed homomorphism between algebras. The Content Addressing Protocol (CAP, named here, future spec) shifts authoring from contracts to citations: authors cite concept CIDs in source, the catalog supplies the contracts, the substrate composes the discharges. The 99.9% of programmers stop writing contracts and start citing concepts. The 0.1% mint new concepts when nothing in the catalog fits. After Static Analysis is the second clause of After Civilization: the chapter where substrate goes from real-for-proof-writers to real-for-ordinary-programmers via citation.

19. **[After Patterns: Concepts as First-Class Language Expression Trees](19-after-patterns-concepts-as-first-class-language-expression-trees.md)**: The cashing-in of three decades of pattern literature. Paper 18 named the `concept:*` hub and the three-key composition that puts it into developer hands; this paper assembles the catalog claim: a concept-citation in source is a first-class language expression tree the substrate expands at realize-time per target language, with the loss-record measuring expressivity per `(concept, language)` cell. The Gang of Four (1994), Effective-X (1996+), POSA (1996+), Refactoring (1999), PoEAA (2002), and Domain-Driven Design (2003) built the substrate's catalog by hand, in English narrative, for thirty years; the substrate corrects the substrate without retiring the work. Pattern discovery becomes free: every algebraic node has a CID, `SELECT cid, COUNT(*) GROUP BY cid ORDER BY count DESC` IS the discovery loop GoF ran by hand, and clustering on the unified algebra finds the patterns no book ever named. The frontier surfaces by exclusion: the cells where every realization carries non-empty loss (concurrent mutable shared state, exact real arithmetic, linearity, dependent types, effect tracking, capability security, typestate, proof-irrelevant witnesses) are the genuine open problems of language design. Karlton's two hard things divide cleanly: cache invalidation dissolves under content-addressing; naming things becomes the durable human contribution, with humans as cartographers of the address space and the substrate enforcing the contracts that get attached to each named beacon. The After-X arc is itself a content-addressed proof-chain: this paper depends on 7, 8, 9, 13, 14, 15, 16, 17, and 18 as lemmas, and the corpus verifies recursively.

20. **[After Hope: The Unification Theorem](20-after-hope.md)**: Closes the After-X arc. The 70-year accident of software's quadratic complexity and the 70-year reign of hope in software engineering are the same architectural fact, viewed from two angles: at every boundary, no shared verifiable claim. When contracts attach to content-addressed concepts in a universal address space, both phenomena terminate at once. The eight-verb pipeline (Lift / Cluster / Name / Scope / Cluster / Identify / Realize / Witness) is the constructive proof. The catastrophe catalogue (Heartbleed, Log4Shell, SolarWinds, Spectre/Meltdown, CrowdStrike, Knight Capital, AWS S3, Cloudflare cloudbleed, SHA-1 collision) names the bill the field has been paying for the gap-blindness. The closing theorem: software ages backwards because the catalog ages forward. Companion to papers 6, 7, 8, 9, 14, 15, 16, 17, 18, 19. Operational artifact: `provekit catalog summarize --include-gaps`.

21. **[After Cross-Language: Every Cross-X Boundary Dissolves the Same Way](21-after-cross-language-every-cross-x-dissolves.md)**: The hidden generalization in papers 12 through 20. The mechanism that dissolved cross-language never used a property of language; it used surface-ness. Lift each surface to ProofIR, name the shared referent with a CID, realize. The same mechanism dissolves cross-library (lodash↔ramda over `concept:deep-equality`), cross-framework (React↔Vue↔Solid over `concept:reactive-cell`), cross-runtime (tokio↔asyncio over `concept:async-task`), cross-architecture (SSE↔NEON↔RISC-V V over `concept:packed-vector-load`), cross-protocol (gRPC↔JSON-RPC over `concept:remote-procedure-call`), cross-database (Postgres↔Mongo over `concept:relational-join`), cross-version (Python 2↔3 over `concept:write-line`), and cross-platform (POSIX↔Win32 over `concept:byte-stream-io`). The math: M+N per axis, additive across axes, not M×N nor product-of-Ms. The concrete consequence: a TypeScript developer contributing to a Zig project without writing Zig is the natural reading of the substrate, not a thought experiment; the realize-pass mediates. The phase transition has a single shape across axes — language becomes accessibility, library becomes sugar, framework becomes render mode, runtime becomes target, architecture becomes effect handler, database becomes query plan, version becomes coordinate. What the mechanism doesn't dissolve: architectural disagreement, performance characteristics, idiomaticity, cultural alignment, domain expertise. The cells of the multi-dimensional catalog get populated by Authored / Inferred / Generated bindings, all discharged into content-addressed claims. Cross-X is one observation; it was hiding inside "cross-language" the whole time.

22. **[After Vendoring: Migration as Source Transformation Over the Program Graph](22-after-vendoring-migration-as-source-transformation.md)**: Paper 21's operational consequence at the cross-library axis. `provekit bind --library-from A --library-to B --write` is not a compatibility layer, adapter, ORM, facade, bridge package, runtime shim, or "common interface": it is a computably correct transformation over the program graph. The migrated repository uses library B directly; library A is removed from the manifest. The hard part of `better-sqlite3` -> `pg` is not the callsite rewrite; it is the call-graph contagion as the `async` effect propagates upstream. The substrate already tracks effect signatures and the call graph is content-addressed, so propagation is mechanical. The deliverable is a signed *receipt* whose first page aggregates the trichotomy into five lines (rewritten / widened / refused / lossy), and whose body cites every changed function with a propagation reason. Refusal and loss-record are first-class outcomes, surfaced at the report tier per the honesty gradient. After the patch lands, the durable artifacts are the patched source and the signed receipt; the substrate is the proof object the receipt cites, not a runtime dependency of the migrated codebase. The substrate exists to compute and justify the patch. Empirical receipt: shipped at `blake3-512:9faa22b5...88910` via PRs #867/#872/#873 (`4 callsites rewritten / 6 widened to async / 1 boundary halt / 1 refused export / 1 lossy site` on the sqlite/pg fixture).

23. **[After Packages: The Proof Envelope Carries the Binding](23-after-packages-the-proof-envelope-carries-the-binding.md)**: Paper 22's operational consequence at the package-distribution tier. A package ships code plus one signed proof envelope; the envelope carries every claim the package makes about itself (contracts, evidence, effects, realizations, `ConceptBinding`s, discharge receipts, signatures) in one admissibility surface. Three rules discriminate the shape: content-addressed is not verified, annotation is not contract, binding claim is not annotation. The binding from a surface op to a concept hub op is a dischargeable semantic claim and lives at the solver-facing tier, not the documentary tier; it belongs INSIDE the proof envelope as a memento, not beside it as a parallel `.sugar` file. (Standalone `.sugar` survives only as optional authoring DSL compiled into mementos at publish time.) The package manifest gains one line (`"provekit": "./provekit.proof"`); the distribution channel gains zero; verify / realize / migrate / audit all read the same envelope filtered by memento kind. Four admission paths (Authored / Self-Attested / Third-party-Inferred / Generated) all produce the same shape: signed mementos in proof envelopes, distributed by npm / PyPI / crates.io / RubyGems / Maven / Hex / NuGet. The substrate inherits the trust infrastructure these registries have accumulated for decades. The catalog is the union of every package author's signed claims, and the catalog has no host.

25. **[The Fire Was Already Hot](25-the-fire-was-already-hot.md)**: The coda of the After-X arc. It withdraws the journey-fiction the prior papers used to teach the substrate: the substrate does not add portability, verification, importability, or proof as new semantic capacity. The capability was latent in the source, package, fixture, and receipt structure all along. Lift recognizes; realize renders; bind routes; prove confirms. The Stage 2 migration receipt, Stage 3 cross-language receipt pair, and witnessed fixture admission read as records of arrival at content-addressed destinations, not as records of a journey. The substrate is anagnorisis, not synthesis. The fire was already hot.

## Contributing a paper

A paper-grade contribution argues a sustained position with engagement of counterarguments, references, and operational consequences. If you have such an argument, file a draft in `docs/papers/draft-NN-<topic>.md` and request review.

Papers are not technical reports; they engage the larger conceptual landscape. They are not blog posts; they engage counterarguments seriously. Both are valuable elsewhere; this directory is for the load-bearing arguments specifically.
