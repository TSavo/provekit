# 2026-05-04 — Dogfood conformance gate catches 8 real bugs in one session

## Headline

The cross-language conformance gate caught 8 real bugs across the day. None would have been visible from local-dev-only testing. Each was exactly the kind of cross-language drift the substrate is designed to catch. The gate caught them in ProvekIt's own house first. Dogfood working as advertised.

## The 8 bugs (chronological)

### 1. csharp test resolver missing built binary

PR #163, first run. `dotnet test` ran without `dotnet build` first; LSP plugin binary missing; 9 LspDaemonProtocolTests failed.

Fix: `test-csharp: build-csharp` Makefile dep at commit `06f0339`.

### 2. maven not on self-hosted runner PATH

PR #163, second run. `make: mvn: No such file or directory`.

Fix: `actions/setup-java@v4` (Temurin 21) at commit `ba1de11`.

### 3. maven multi-module dep resolution failed

PR #163, third run. `provekit-lift-java-core` couldn't resolve sibling `provekit-ir`.

Fix: switch to parent pom + `-pl provekit-lift-java-core -am` + `mvn install` at commit `9627a95`.

### 4. C3 contract violation in cmd_mint::dispatch

Surfaced by PR #165 conformance gate agent (a9a50c321df0e4a9d). `source_paths: []` in lift-protocol RPC violates `verify_c3_lift_request_well_formed`. The dogfood gate caught the substrate violating its own canonical contracts.

Fix: `source_paths: ["."]` at commit `c5d61a7` (PR #169).

### 5. z3 SMT classifier silently returning "undecidable"

Surfaced by PR #165's prove-* matrix. `spawn("z3", ...)` returned ENOENT in CI; try/catch reified as "unknown"; `classifyVerdict` cascaded to "undecidable" for every implication. 7 tests across 3 files affected.

Fix: switch to z3-solver WebAssembly (`z3-solver@4.16.0` npm pkg) at commit `9065739` (PR #175).

### 6. KIT_TABLE pointing at test-fixture lifters, not self-contracts surfaces

Surfaced by go canonical-mirror agent (a0eed683ff30b42fc). `--kit=go` resolved to surface `go` (test-fixture lifter, static empty proof envelope). Made architecture decision (c) on issue #176; implemented in PRs #180 (go) + #183 (rust). KIT_TABLE expanded to 4-tuple, kit alias and lift surface decoupled.

Empirical contractSetCid drift: rust `26814dfc...` to `8f4bcc3c...`, go `d53d18c2...` to `e23649f3...`. First time the dogfood is content-meaningful for those two kits.

### 7. mint_kit_integration test race on rust.json

Surfaced by PR #183 conformance gate. Multiple tests writing to same attestation file concurrently, causing EOF parse panic. Pre-existing, but only became visible once #180 + #183 made the tests do real work.

Fix: `serial_test 3.4.0` named lock `mint_kit_files` at commit `0c68c19` (PR #187).

### 8. v1.4.1 catalog drift between PR #178 (interim CID) and PR #179 (real CID)

PR #178 bumped Makefile to `9cb8600c...` based on the catalog signature file at the time; PR #179 revealed that was a stale interim version. Real v1.4.1 catalog CID is `dc2f42ff...`. Caught by the docs sweep agent (PR #184) verifying zero `9cb8600c` references survive.

Filed as #181 for re-stamp.

## Architectural lessons

**The gate is doing what the gate is for.** Cross-kit drift is exactly the failure class the substrate exists to catch. ProvekIt's own conformance gate catching ProvekIt's own bugs is the dogfood working at full strength.

**The 3-5 minute gate runtime is a feature, not a defect.** The fix-and-restart loop is what's actually expensive when a bug ships; the gate itself is cheap.

**Architectural option (c) was empirically refined to (b)+(c) hybrid.** The strict-(c) hypothesis ("same lifter walks self-contracts") was empirically false in practice. `provekit-lift` and `mint-self-contracts` are distinct binaries producing distinct CIDs. The correct cut is two surfaces per kit (general-purpose + self-contracts), with KIT_TABLE routing `--kit=<lang>` to the self-contracts surface as the canonical "what does this kit attest about itself" answer. See issue #176 comment thread.

**Test isolation is its own substrate concern.** Integration tests sharing repo-level state were correct integration coverage, but parallel execution turned shared state into races. The fix is `serial_test` named locks scoped per file, not architectural redesign. These tests SHOULD exercise real repo state.
