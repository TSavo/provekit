## v0.5 whitelist extension (this PR)

The v0 expression whitelist was the dominant skip cause in the original
self-lift (53/69 = 77%). v0.5 admits the canonicalizer's idiomatic
shape without expanding to "every Rust expression". Newly-accepted
operand shapes:

  - free-function call `f(a, b, ...)`           -> Ctor("f", [args])
    (was: single-arg only)
  - method call `recv.f(a, b)`                  -> Ctor("f", [recv, a, b])
    (UFCS-flatten one level; was: rejected outright)
  - reference `&expr`                           -> lift(expr) (transparent)
  - cast `expr as T`                            -> lift(expr) (transparent)
  - array literal `[a, b]`                      -> Ctor("array", [args])
  - tuple literal `(a, b)`                      -> Ctor("tuple", [args])
  - vec! macro `vec![a, b]`                     -> Ctor("vec", [args])
  - byte-string literal `b"..."`                -> Ctor("byte_str", [str])
  - binary op `a OP b` in operand position      -> Ctor("<op>", [a, b])

Out of scope (intentionally): every Rust expression, the assert! body
predicate grammar (assert! still requires a binary comparison), pattern
expressions beyond the existing `assert_matches!` whitelist, format!
and other macro calls in operand position, indexing, field access,
closures, blocks, `.await`, ranges, type ascription.

## Findings

  1. The v0.5 whitelist extension closed the dominant skip category.
     The 53-hit "expression shape outside v0 whitelist" bucket reduced
     to 1-2 residual hits in v0.5. The 14-hit "assert! body not a
     comparison" bucket was kept out of scope (separate code path);
     the 2-hit byte-string literal bucket was addressed (+2 lifts).

  2. Spec-level claims that v0 could not reach are now lifted: sorted
     object keys (multiple), control-character escapes, integer plain
     decimal, boolean/null literals,
     distinct-inputs-distinct-hashes via b"". The byte-faithful
     equality of an `encode_jcs` result against a literal canonical
     string is now the predominant lifted shape in this crate's
     catalog.

  3. The lift is structurally faithful, not semantically reductive. A
     test like
         assert_eq!(encode_jcs(&Value::object([("a", "x")])), "{\"a\":\"x\"}")
     produces an IR atom whose LHS is a fully-nested
         Ctor("encode_jcs", [Ctor("Value::object", [Ctor("array",
             [Ctor("tuple", [str("a"), str("x")])])])])
     and whose RHS is the literal canonical string. The verifier may
     not yet have a semantic interpretation of `encode_jcs` as a
     primitive (no bridge), but the lattice is clean and the
     predicate is content-addressable.

  4. Idempotency holds: repeated `make self-lift-canonicalizer`
     against the same source produces the same catalog CID
     blake3-512:2f78c6d6... regardless of run order. The .proof file
     is replaced in place; the old v0 CID file is removed by the
     target's `rm -f` step.

  5. The pre-existing `jcs.invariant.rs` and `hash.invariant.rs` in
     this crate hand-author the same shape of claims the v0 lifter
     could not reach. v0.5 brings the auto-lift path much closer to
     the hand-author path on this crate; the auto-lifted catalog now
     covers most of what the hand-authored invariant files express on
     the spec-level claims (sorted keys, escape sequences, integer
     formatting, distinct hashes). The residual is
     `assert!`-with-method-result-as-bool predicates (`starts_with`,
     `matches!(...)`) which need either a v0.6 predicate-grammar
     extension or an IR-level atom.
